## Architecture Stress Tester (gpt-5.3-codex)

### Findings

**[CRITICAL] Single mutable JSON cache is a write-time SPOF with no concurrency control**
- Location: `scripts/cto_context_search.py` (refresh-cache block) + Verification section
- Issue: The system has one shared cache file that is rebuilt wholesale, so concurrent refresh/search processes can lose updates or serve inconsistent state.
- Evidence: `"cache = {}  # Start fresh"` and later `"save_embedding_cache(cache)"`; plan also hard-codes `"Cache location: logs/cache/vault_embeddings.json"`.
- Recommendation: Replace JSON cache with a transactional store (SQLite/DuckDB), add file/process locking, and use generation IDs (write new generation, then atomic pointer switch) so readers never see partial or stale writes.

**[HIGH] Embedding refresh pipeline is strictly sequential and will not hold at 10x/100x corpus growth**
- Location: `scripts/cto_context_search.py` (refresh-cache block), Phase 3: Incremental Refresh
- Issue: Embeddings are generated one file at a time via synchronous calls, creating linear refresh latency and backlog as corpus size grows.
- Evidence: `"for i, filepath in enumerate(learning_files, 1): ... embedding = embed_text(text)"`; plan target says `"Cache refresh completes in <5 seconds for typical delta"`.
- Recommendation: Add a background indexing worker with bounded parallelism and batching, persist per-file job status, and separate query path from indexing path so search latency is not coupled to refresh operations.

**[HIGH] Semantic and keyword corpora are implicitly divergent (distributed-monolith behavior at retrieval layer)**
- Location: `scripts/cto_context_search.py` + `scripts/cto_context_lib.py` + Phase 2: Search flow
- Issue: Semantic refresh indexes only `knowledge/learnings`, while keyword search spans multiple weighted repositories, producing inconsistent and non-comparable results.
- Evidence: `"learnings_path = ROOT / 'knowledge' / 'learnings'"` vs `SOURCE_WEIGHTS = { "02-validated/patterns/context-packs", "fundamentals", "knowledge/learnings", "knowledge/answers", "01-active", "logs/transcripts" }`.
- Recommendation: Unify corpus selection behind a shared index policy (same prefixes for both paths), or explicitly expose per-index scopes and return scope metadata so users understand coverage differences.

**[HIGH] Stale/deleted content lifecycle is under-specified, risking ghost results and broken references**
- Location: Phase 1: Cache management / Phase 3: Smart caching
- Issue: Cache freshness is defined around modified mtimes only, with no explicit deletion/rename reconciliation.
- Evidence: `"get_stale_files(cache): # Compare mtimes, return files needing re-embed"` and `"Smart caching — Only re-embed files modified since last cache"`.
- Recommendation: On each refresh, diff cache keys against live file inventory; remove missing paths, handle renames, and validate file existence before returning semantic hits.

**[MEDIUM] Model-version drift creates hidden coupling and non-reproducible retrieval quality**
- Location: Prerequisites + Ollama API + Verification
- Issue: Embeddings depend on a local model name without pinning digest/version, so results can drift silently across machines and time.
- Evidence: `"nomic-embed-text model | ollama pull nomic-embed-text"` and cache stores embeddings without documented model fingerprint.
- Recommendation: Store `model_name`, `model_digest`, and embedding schema version in cache metadata; auto-invalidate/rebuild on mismatch.

**[MEDIUM] Query path is full-scan vector ranking over JSON; this is under-engineered for 100x**
- Location: Architecture + Verification
- Issue: Current design implies loading and scoring all vectors per query from JSON, which will drive latency/memory spikes as corpus grows.
- Evidence: `"Embedding Cache (JSON)"`, `"Cache size: ~2.0MB"` for `"99 learnings embedded (768 dimensions each)"`.
- Recommendation: Move to ANN-capable indexing (FAISS/HNSW or SQLite extension), precompute normalized vectors, and enforce latency SLOs (p50/p95) with load tests at 1k/10k/100k docs.

**[MEDIUM] Output path is a dual list (semantic + keyword) with no rank fusion, creating operator bottlenecks**
- Location: `scripts/cto_context_search.py` (semantic mode output), Phase 2: Display format
- Issue: Returning two independent ranked lists forces manual reconciliation and increases decision latency as result counts grow.
- Evidence: Code emits separate blocks: `"## Semantic Results..."` and `"## Keyword Results"`; JSON output has separate `"semantic"` and `"keyword"` arrays.
- Recommendation: Implement reciprocal-rank fusion (or weighted blend) with deduplication and provenance tags; keep optional split view for debugging only.

**[LOW] Failure handling degrades silently, masking service health issues**
- Location: Error Handling + `scripts/cto_context_search.py`
- Issue: Automatic keyword fallback preserves availability but hides semantic outage frequency and impact.
- Evidence: `"Warning: Ollama not available. Falling back to keyword search."` and plan note `"Do NOT require Ollama as hard dependency"`.
- Recommendation: Add telemetry counters/events (`semantic_requested`, `semantic_succeeded`, `fallback_reason`) and optional `--require-semantic` mode for CI/benchmark runs.

### Top 3 Recommendations

1. Replace `vault_embeddings.json` with a transactional, lock-safe index store and generation-based reads/writes (Finding: **Single mutable JSON cache is a write-time SPOF**).
2. Decouple indexing from query serving with asynchronous/batched embedding jobs and bounded parallelism (Finding: **Embedding refresh pipeline is strictly sequential**).
3. Unify corpus scope and lifecycle rules (coverage + deletion handling) across semantic and keyword paths (Findings: **Semantic/keyword corpus divergence** and **stale/deleted lifecycle gap**).

### What I Might Be Wrong About

- If this tool is strictly single-user, single-process CLI usage, I may be overstating the concurrency/cache corruption severity; multi-process contention risk drops sharply.
- I’m assuming `search_semantic()` does full-scan over the cache; if it already uses pre-normalized vectors and efficient in-memory reuse, the 100x latency risk is lower.
- I rated corpus divergence as HIGH because it causes inconsistent retrieval contracts; if intentional (semantic only for learnings by design), severity is closer to MEDIUM with better UX labeling.

## Deployment Risk Assessor (deepseek-v3.2)

### Findings

**[CRITICAL] No Rollback Path for Cache Corruption or Schema Changes**
- Location: Phase 1: Embedding Infrastructure, Phase 3: Incremental Refresh
- Issue: The plan creates a new, critical data store (`vault_embeddings.json`) with no defined procedure for reverting it to a known-good state if the cache becomes corrupted, the embedding model changes, or the schema evolves.
- Evidence: "Cache management — `logs/cache/vault_embeddings.json`" and "Smart caching — Only re-embed files modified since last cache". The plan mentions "Cache corrupt | Rebuild from scratch" in error handling but provides no mechanism to trigger this safely or restore from a backup.
- Recommendation: Implement a versioned backup of the cache file before any write operation (e.g., using the existing `backup_before_write` utility). Add a `--restore-cache` CLI flag to revert to the last known-good backup. Document the cache schema version and include a validation step on load.

**[HIGH] Blast Radius of Cache Staleness Affects All Semantic Search Users**
- Location: Phase 3: Incremental Refresh, Implementation Details: Embedding Target
- Issue: If the incremental refresh logic (`get_stale_files`) fails silently or incorrectly determines a file is unchanged, all users of semantic search will receive stale, potentially irrelevant results until a full cache refresh is manually triggered.
- Evidence: "Smart caching — Only re-embed files modified since last cache" and "Cache TTL — Configurable (default: 24 hours)". The failure of this single component degrades the core functionality of the entire upgrade for all users.
- Recommendation: Implement a "stale-but-usable" warning system. If a file's mtime is newer than the cache entry, log a warning and optionally serve the stale result while queuing a background re-embed. Add a health check endpoint or CLI command that reports cache freshness.

**[MEDIUM] No Backward Compatibility for Search Clients**
- Location: Phase 2: Semantic Search Implementation
- Issue: The `--semantic` flag and the new hybrid JSON output format are additive, but there is no guarantee that existing scripts or users parsing the keyword-only JSON output will be compatible with the new structure, potentially breaking integrations.
- Evidence: "Display format (add to existing)" and "JSON output includes both semantic and keyword results". The plan changes the output schema when `--semantic` is used without confirming if callers expect the old format.
- Recommendation: Maintain strict backward compatibility for the default (keyword-only) path. For the `--semantic` path, provide the new hybrid output *in addition to* the legacy format via a `--legacy-json` flag, or clearly document the breaking change and version the API.

**[MEDIUM] Deployment Sequence Lacks Verification and Staging Step**
- Location: Execution Sequence
- Issue: The plan moves directly from implementation to production use in the vault without a staged rollout or verification that the new semantic search provides equal or better correctness compared to the existing keyword search.
- Evidence: "Week 2 └── Validation: 10-query benchmark vs keyword". Validation occurs *after* the implementation is considered complete and integrated. A faulty embedding function could degrade search quality silently.
- Recommendation: Insert a "Phase 0: Shadow Evaluation" where semantic search runs in parallel but its results are logged and compared to keyword search for a period (e.g., 1 week) before being exposed to users via the `--semantic` flag. Use the Evaluation Criteria table to gate the release.

**[LOW] Point of No Return for Ollama Model Not Documented**
- Location: Prerequisites, Architecture
- Issue: The plan depends on the `nomic-embed-text` model. If this model is deprecated, changed, or produces different vector dimensions, the entire cache becomes invalid. The point where switching models requires a full, irreversible cache rebuild is not acknowledged.
- Evidence: "nomic-embed-text model | `ollama pull nomic-embed-text` | Model downloaded" and `embed_text(text: str, model: str = "nomic-embed-text")`. The model name is hard-coded as a default.
- Recommendation: Store the model name and expected embedding dimension within the cache metadata (`vault_embeddings.json`). Validate these on cache load. If they don't match the current runtime configuration, invalidate the cache with a clear error message instructing a full `--refresh-cache`.

### Top 3 Recommendations

1. **[CRITICAL Fix]** Before deploying, implement a rollback and recovery mechanism for the `vault_embeddings.json` cache, including versioned backups and a restore command. (Relates to: CRITICAL - No Rollback Path for Cache Corruption)
2. **[HIGH Fix]** Add robustness to the cache freshness logic: implement warnings for stale entries and a health check to proactively identify degradation. (Relates to: HIGH - Blast Radius of Cache Staleness)
3. **[MEDIUM Fix]** Run a shadow evaluation phase where semantic search results are collected and compared to keyword baselines before enabling the `--semantic` flag for general use. (Relates to: MEDIUM - Deployment Sequence Lacks Verification)

### What I Might Be Wrong About

- **Assumption about integration breakage:** I assumed there are external scripts or systems parsing the JSON output of `cto_context_search.py`. If this tool is only used interactively by humans via CLI, the backward compatibility concern is LOW, not MEDIUM.
- **Lack of context on failure impact:** I lack context on how critical the semantic search functionality is to daily operations. If it's a "nice-to-have" enhancement on top of a working keyword system, the blast radius of cache issues is smaller, and the HIGH finding might be MEDIUM.
- **Severity of model dependency:** I rated the model dependency as LOW. However, if the `nomic-embed-text` model is unstable or frequently updated in a breaking way, this could become a HIGH risk, causing recurrent cache invalidations and operational overhead.

## Observability Advocate (gemini-3.1-pro)

### Findings

**[CRITICAL] Silent Degradation on Embedding Failures**
- Location: Error Handling
- Issue: Skipping files when embedding fails without structured error logging or summary metrics means users will silently lose semantic search coverage for specific files without knowing it.
- Evidence: "Embedding fails | Skip file, continue with others"
- Recommendation: Implement a summary report at the end of `--refresh-cache` that outputs the exact count of failed files, and log the specific file paths and HTTP error codes to a dedicated `logs/embedding_errors.log` file.

**[HIGH] Unbounded JSON Cache Loading Latency**
- Location: Phase 1: Embedding Infrastructure
- Issue: Parsing a monolithic JSON file into memory on every synchronous CLI execution will severely degrade search latency as the vault grows, violating the saturation golden signal.
- Evidence: "Monitor cache size; 100 learnings × 768 dims ≈ 300KB uncompressed (actual: ~2MB with JSON overhead)"
- Recommendation: Define a hard threshold for cache size (e.g., 10MB) that triggers a console warning, and plan a migration to a local embedded database (e.g., SQLite with vector extensions) to avoid full-file JSON parsing on every search.

**[HIGH] Missing Timeout Controls for Ollama API**
- Location: Implementation Details / Ollama API
- Issue: There are no timeouts specified for the synchronous HTTP calls to the local Ollama instance, meaning a stalled Ollama process will cause the CLI to hang indefinitely instead of falling back.
- Evidence: `curl http://localhost:11434/api/embeddings -d '{ "model": "nomic-embed-text", "prompt": "Your text here" }'`
- Recommendation: Enforce a strict HTTP timeout (e.g., 2 seconds for search, 10 seconds for cache refresh) on the Ollama API call, and catch `TimeoutError` to explicitly trigger the graceful keyword fallback.

**[MEDIUM] Lack of Trace Context to Local LLM**
- Location: Implementation Details / Ollama API
- Issue: HTTP requests to Ollama lack identifying headers, making it impossible to distinguish vault queries from other local agent activities in Ollama's server logs during an incident.
- Evidence: `curl http://localhost:11434/api/embeddings -d '{...}'`
- Recommendation: Inject a custom header (e.g., `User-Agent: cto-vault-search/1.0`) into the Ollama HTTP request to allow request correlation in the Ollama server logs.

**[MEDIUM] Ambiguous Cache Corruption Detection**
- Location: Error Handling
- Issue: The plan specifies rebuilding the cache if it is corrupt, but does not define what constitutes corruption or how it is detected, leading to potential crash loops if the file is locked or partially written.
- Evidence: "Cache corrupt | Rebuild from scratch"
- Recommendation: Explicitly catch `json.decoder.JSONDecodeError` during `load_embedding_cache()`, move the corrupted file to `logs/cache/vault_embeddings.json.corrupt`, and initialize an empty cache dictionary to force a clean rebuild.

### Top 3 Recommendations

1. Add strict HTTP timeouts and explicit `TimeoutError` handling for all Ollama API requests to prevent silent hangs and ensure the keyword fallback actually executes when the model stalls.
2. Implement explicit error logging and a final failure summary during `--refresh-cache` so users know exactly which files are missing from the semantic index.
3. Replace the monolithic JSON cache with a scalable local store (e.g., SQLite) before the JSON parsing overhead degrades CLI latency beyond acceptable interactive limits.

### What I Might Be Wrong About

- I assumed the vault will grow significantly beyond 100 learnings. If this is a strictly bounded personal vault, a 2MB-5MB JSON parse might remain sub-50ms and never become a real latency issue.
- I lack context on whether Ollama is running as a systemd service with its own robust logging that the user already monitors, which might mitigate the need for client-side trace headers.
- The lack of trace context (Finding 4) might be less severe than I rated it (MEDIUM) if this is a single-user local environment where concurrent Ollama requests from different applications are extremely rare.

## Coupling Detector (qwen-3.5-27b)

### Findings

**[CRITICAL] Missing backup_before_write usage in cache operations**
- Location: `scripts/cto_context_search.py` (lines 35-55)
- Issue: The `backup_before_write` function exists in `cto_context_lib.py` but is never called when `save_embedding_cache` writes to the cache file, violating the plan's own "Shadow Restore Points" pattern
- Evidence: `scripts/cto_context_lib.py` defines `backup_before_write(filepath: str | Path) -> Optional[Path]` but `scripts/cto_context_search.py` calls `save_embedding_cache(cache)` directly without backup
- Recommendation: Wrap `save_embedding_cache` calls with `backup_before_write(BACKUP_DIR / "vault_embeddings.json")` before any cache write operation

**[HIGH] Import statement inside conditional block breaks module-level dependency tracking**
- Location: `scripts/cto_context_search.py` (lines 35-40)
- Issue: Functions are imported inside the `if '--refresh-cache' in sys.argv:` block, creating hidden dependencies that static analysis tools cannot detect
- Evidence: `from cto_context_lib import (load_embedding_cache, save_embedding_cache, _extract_learning_text, embed_text, ROOT)` appears inside conditional, not at module top
- Recommendation: Move all imports to module level; use `if args.refresh_cache:` after parsing instead of checking `sys.argv` directly

**[HIGH] Cache file has no existence check before load**
- Location: `scripts/cto_context_search.py` (line 42)
- Issue: `cache = {}` starts fresh on refresh, but `load_embedding_cache()` is called without checking if cache file exists, risking FileNotFoundError on first run
- Evidence: Plan states "Cache corrupt: Rebuild from scratch" but no try/except around cache loading is visible in the code
- Recommendation: Add `try: cache = load_embedding_cache() except FileNotFoundError: cache = {}` pattern to handle missing cache gracefully

**[MEDIUM] search_semantic function signature not visible in truncated code**
- Location: `scripts/cto_context_lib.py` (truncated at line 795)
- Issue: The plan claims `search_semantic()` was implemented (lines 532-700) but the provided code is truncated, making interface verification impossible

**[MEDIUM] Large cache file (89,012 lines) may cause I/O bottlenecks**
- Location: `logs/cache/vault_embeddings.json`
- Issue: The cache file is ~2MB with 99 embeddings at 768 dimensions each; full JSON serialization/deserialization on every search could be slow
- Evidence: File has 89,012 lines; plan states "Cache size: ~2.0MB" but no compression or binary format is mentioned
- Recommendation: Consider using `pickle` or `msgpack` for binary serialization, or implement lazy loading that only reads needed embeddings

**[MEDIUM] Temporal coupling: Ollama availability check happens after cache refresh starts**
- Location: `scripts/cto_context_search.py` (lines 35-36)
- Issue: The `--refresh-cache` flow checks `ollama_available()` but then proceeds to embed all files without verifying Ollama remains available during the operation
- Evidence: `if not ollama_available(): print("Error: Ollama not available. Cannot refresh cache.")` but no re-check during the embedding loop
- Recommendation: Add `ollama_available()` check inside the embedding loop with early exit on failure

**[LOW] SOURCE_WEIGHTS global couples search behavior to hardcoded paths**
- Location: `scripts/cto_context_lib.py` (lines 33-40)
- Issue: The `SOURCE_WEIGHTS` dict hardcodes directory paths that must match the actual vault structure; changing vault organization requires code changes
- Evidence: `SOURCE_WEIGHTS = {"02-validated/patterns/context-packs": 6, "fundamentals": 5, ...}` is a global constant
- Recommendation: Move `SOURCE_WEIGHTS` to a config file or make it configurable via environment variable

### Top 3 Recommendations

1. **Add backup_before_write to cache operations** (relates to CRITICAL finding) — Wrap all `save_embedding_cache` calls with backup logic to enable safe cache recovery
2. **Move imports to module level** (relates to HIGH finding) — Refactor `scripts/cto_context_search.py` to use standard import patterns for better static analysis and dependency tracking
3. **Add cache existence check** (relates to HIGH finding) — Implement try/except around `load_embedding_cache()` to handle first-run scenarios gracefully

### What I Might Be Wrong About

- **The search_semantic implementation may be complete** — The code was truncated at line 795, but the plan claims implementation is complete. If the full file shows proper error handling and interface stability, the MEDIUM severity findings may be overstated
- **Cache performance may be acceptable** — I assumed 89,012 lines would cause I/O issues, but if the system runs on SSD with adequate RAM, JSON serialization might be fast enough for the use case
- **The backup_before_write pattern may be intentionally scoped** — The plan mentions "agent-level undo, not git-level" for backups. The cache might be considered disposable compared to knowledge files, making the missing backup less critical than I rated
