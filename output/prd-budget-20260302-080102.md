## Requirements Archaeologist (gemini-3-flash)

### Findings

**[CRITICAL] Silent Cache Invalidation on Model Change**
- Location: Phase 1: Embedding Infrastructure / `scripts/cto_context_lib.py`
- Issue: The cache does not store the model name or dimension count, leading to corrupted similarity scores if the user switches Ollama models (e.g., from `nomic-embed-text` to `llama3`).
- Evidence: `cache[rel_path] = {"embedding": embedding, "mtime": filepath.stat().st_mtime, ...}`
- Recommendation: Add a `metadata` key to `vault_embeddings.json` storing the `model_name`. If the current model doesn't match the cache metadata, force a full refresh or maintain model-specific cache buckets.

**[HIGH] Unbounded Memory Consumption during Search**
- Location: Phase 2: Semantic Search Implementation / `search_semantic`
- Issue: The plan loads the entire embedding cache into memory and performs a linear scan for every query, which will cause latency spikes and OOM risks as the vault grows.
- Evidence: `cache = {} # Start fresh ... save_embedding_cache(cache)` and `99 learnings embedded ... Cache size: ~2.0MB`
- Recommendation: Implement a maximum vault size limit for the linear scan or transition to a lightweight vector index (like `faiss-cpu` or a simple binary search on normalized vectors) once the cache exceeds 50MB.

**[HIGH] Race Condition in Atomic Cache Writes**
- Location: Phase 1: Embedding Infrastructure / `save_embedding_cache`
- Issue: The plan specifies "Atomic write," but standard Python `json.dump` to a file is not atomic and can result in a truncated/corrupt cache if the process is interrupted.
- Evidence: `save_embedding_cache(cache) # Atomic write to cache file`
- Recommendation: Write the JSON to a temporary file (e.g., `vault_embeddings.json.tmp`) and use `os.replace()` to overwrite the original file, ensuring the cache is never in a partially-written state.

**[MEDIUM] Missing Character/Token Limit on Embedding Inputs**
- Location: Implementation Details / Embedding Target
- Issue: Concatenating Title, Thesis, and Tags without a length check may exceed the context window of the embedding model (typically 2048-8192 tokens), leading to silent truncation of the most important data (Tags).
- Evidence: `Concatenate: title + " | " + thesis + " | " + tags`
- Recommendation: Implement a hard character limit (e.g., 2000 chars) for the concatenated string and log a warning if a learning is truncated before the Tags are included.

**[MEDIUM] Implicit Dependency on Localhost Networking**
- Location: Implementation Details / Ollama API
- Issue: The plan assumes Ollama is bound to `localhost:11434`, which fails in containerized environments (Docker/Podman) or remote dev-server setups.
- Evidence: `POST to localhost:11434/api/embeddings`
- Recommendation: Use an environment variable (e.g., `OLLAMA_HOST`) with a default value of `localhost:11434` to allow configuration in non-standard environments.

**[LOW] Brittle Heading Extraction**
- Location: Implementation Details / Embedding Target
- Issue: The plan assumes the first `# ` is the title, but many markdown files use frontmatter or multiple H1s, leading to "noisy" embeddings.
- Evidence: `Title (first # heading)`
- Recommendation: Fall back to the filename if no `# ` heading is found, and strip Markdown syntax (links, bolding) from the title before embedding.

### Top 3 Recommendations

1. **Implement Model-Versioning in Cache**: (Relates to CRITICAL finding) Prevent mathematical "garbage-in-garbage-out" by ensuring embeddings in the cache were generated by the same model currently active in Ollama.
2. **Ensure Atomic File Operations**: (Relates to HIGH finding) Use the `write-to-temp-and-move` pattern for `vault_embeddings.json` to prevent the "Corrupt Cache" failure mode identified in the Error Handling table.
3. **Add Input Sanitization/Truncation**: (Relates to MEDIUM finding) Explicitly bound the string length sent to the `/api/embeddings` endpoint to ensure the "Tags" (which are high-signal for search) aren't dropped by the model's internal token limits.

### What I Might Be Wrong About

- **Linear Scan Performance**: I rated the linear scan as a HIGH risk, but if the vault is intended to stay under 1,000 files, the 768-dimension dot product in NumPy might remain sub-100ms for years, making a vector DB unnecessary.
- **Ollama API Stability**: I assumed the `/api/embeddings` endpoint is stable; if Ollama changes their API schema (as they are in active development), the library will break regardless of the Python implementation.
- **Nomic-Embed-Text Context Window**: I assumed a standard context window; if this specific model has an unusually small window (e.g., 512 tokens), the truncation risk is much higher than "MEDIUM."

## Testability Auditor (gemini-3-flash)

### Findings

**[HIGH] Undefined Similarity Threshold for "Match"**
- Location: Phase 2: Semantic Search Implementation
- Issue: The specification lacks a minimum cosine similarity score to qualify a result as a "match," leading to non-deterministic result sets.
- Evidence: "Return top-5 by cosine similarity + scores" and "Return similarity score 0-1"
- Recommendation: Define a static `SIMILARITY_THRESHOLD` (e.g., 0.70). The test should verify that results with scores below this threshold are excluded, even if fewer than 5 results exist.

**[MEDIUM] Ambiguous "Thesis Line" Extraction Logic**
- Location: Implementation Details > Embedding Target
- Issue: The definition of "thesis line" is structurally loose, making the input data for embeddings non-deterministic across different Markdown styles.
- Evidence: "Thesis line (first paragraph after title)"
- Recommendation: Define "thesis line" as the first non-empty block following the H1 header that does not start with a Markdown metadata pattern (like `**Date**:` or `**Source**:`).

**[MEDIUM] Missing Cache Collision/Stale Data Strategy**
- Location: Phase 1: Embedding Infrastructure
- Issue: The cache uses `filepath` as a key but doesn't specify behavior if a file is renamed, potentially leading to orphaned embeddings or duplicate entries for the same content.
- Evidence: "Returns {filepath: {"embedding": [...], "mtime": ...}}"
- Recommendation: Use a content hash (SHA-256) in addition to `mtime` to validate cache entries, and implement a cleanup routine in `save_embedding_cache` to remove keys no longer present on disk.

**[LOW] Non-Deterministic "Top 5" Tie-Breaking**
- Location: Phase 2: Semantic Search Implementation
- Issue: If multiple files return the exact same cosine similarity score, the sort order is undefined.
- Evidence: "Return top-5 by cosine similarity + scores"
- Recommendation: Specify a secondary sort key (e.g., file modification time descending or alphabetical path) to ensure consistent test output.

**[CRITICAL] Untestable "Token Reduction" Criterion**
- Location: Evaluation Criteria
- Issue: The target "Find learnings keyword misses" is a subjective quality metric that cannot be verified with a binary Pass/Fail test without a pre-defined "Golden Set" of queries and expected missing documents.
- Evidence: "Token reduction | Compare semantic vs keyword for 10 queries | Find learnings keyword misses"
- Recommendation: Create a "Golden Query Set" containing 10 specific queries and the exact file paths they *must* return via semantic search which keyword search is known to fail on.

### Top 3 Recommendations

1. **Establish a Golden Data Set (Relates to CRITICAL finding)**: Create a static fixture of 10 queries and their expected semantic matches. Without this, "better than keyword" is a subjective opinion, not a verified requirement.
2. **Hard-code the Similarity Floor (Relates to HIGH finding)**: Implement a `MIN_SEMANTIC_SCORE` constant. Testing "Top 5" without a floor results in "hallucinated" context where the 5th result might have a 0.2 similarity and be completely irrelevant.
3. **Formalize the Embedding String Template (Relates to MEDIUM finding)**: Replace the "if present" logic with a strict template function. If a field (like Tags) is missing, it must be replaced with a deterministic empty string constant to ensure embedding consistency.

### What I Might Be Wrong About

- **Numpy Dependency**: I assumed `numpy` is required for cosine similarity based on the "Prerequisites" table, but the implementation could technically be done in pure Python. If pure Python is used, performance for 100+ files might degrade, changing the "Efficiency" test targets.
- **Ollama API Latency**: I am assuming the local Ollama API responds within a timeframe that allows for "incremental refresh < 5s." If the model load time or inference is slower on target hardware, the "Success Metrics" for cache refresh are unattainable.
- **Markdown Consistency**: I assumed all vault files follow the `# Title` format. If the vault contains files using `Title\n===` (Setext-style headers), the current extraction logic for embeddings will fail to find the title, resulting in "empty" embeddings.

## Edge Case Hunter (qwen-3.5-27b)

### Findings

**[CRITICAL] Path Traversal in Backup Function**
- Location: `scripts/cto_context_lib.py` - `backup_before_write()`
- Issue: The function accepts arbitrary filepath strings without validation, allowing attackers to backup files outside the intended vault directory.
- Evidence: `def backup_before_write(filepath: str | Path) -> Optional[Path]:` with no path validation or canonicalization
- Recommendation: Add `Path(filepath).resolve()` and verify it starts with `ROOT` before proceeding. Reject paths containing `..` or absolute paths outside the vault.

**[CRITICAL] Unbounded Query Input Enables DoS**
- Location: `scripts/cto_context_search.py` - `--query` argument
- Issue: No maximum length specified for query strings, allowing resource exhaustion through extremely long queries.
- Evidence: `parser.add_argument("--query", required=True, help="Query string.")` with no `max_length` or validation
- Recommendation: Add `max_length=1000` constraint and validate query length before processing. Reject queries exceeding limit with clear error message.

**[HIGH] No Maximum Limit on Search Results**
- Location: `scripts/cto_context_search.py` - `--limit` argument
- Issue: The `--limit` parameter has no upper bound, allowing attackers to force processing of all vault files and exhaust memory/CPU.
- Evidence: `parser.add_argument("--limit", type=int, default=15, help="Max matches.")` with no `choices` or validation
- Recommendation: Add `choices=range(1, 101)` or validate `limit <= 100` in code. Document the maximum in help text.

**[HIGH] Cache File Race Condition (TOCTOU)**
- Location: `scripts/cto_context_lib.py` - `get_stale_files()` and `save_embedding_cache()`
- Issue: mtime comparison for cache freshness is vulnerable to TOCTOU attacks; file could be modified between check and embed.
- Evidence: `def get_stale_files(cache: Dict) -> List[Path]:` compares mtimes without file locking
- Recommendation: Use file locking during cache operations. Consider using `os.flock()` or atomic file operations with temp file + rename pattern.

**[HIGH] Non-Atomic Cache Write**
- Location: `scripts/cto_context_lib.py` - `save_embedding_cache()`
- Issue: Spec claims "Atomic write" but implementation not shown; partial writes could corrupt cache.
- Evidence: `def save_embedding_cache(cache: Dict): # Atomic write to cache file` - comment only, no implementation
- Recommendation: Implement atomic write using `tempfile.mkstemp()` followed by `os.replace()` to ensure cache integrity.

**[MEDIUM] No Embedding Dimension Validation**
- Location: `scripts/cto_context_lib.py` - `embed_text()`
- Issue: No validation that Ollama returns expected 768-dimension vectors; malformed responses could cause crashes.
- Evidence: `def embed_text(text: str, model: str = "nomic-embed-text") -> List[float]:` with no dimension check
- Recommendation: Validate `len(embedding) == 768` before returning. Log warning and skip file if dimension mismatch.

**[MEDIUM] Unvalidated Model Parameter**
- Location: `scripts/cto_context_lib.py` - `embed_text()`
- Issue: Model name is not validated, allowing injection of unexpected model names or potential API abuse.
- Evidence: `def embed_text(text: str, model: str = "nomic-embed-text")` accepts arbitrary string
- Recommendation: Whitelist allowed models: `if model not in ["nomic-embed-text"]: raise ValueError("Invalid model")`

**[MEDIUM] No Cache Integrity Verification**
- Location: `logs/cache/vault_embeddings.json`
- Issue: No checksum or validation when loading cache; corrupted cache could cause silent failures.
- Evidence: `def load_embedding_cache() -> Dict[str, Dict]:` with no integrity check
- Recommendation: Add JSON schema validation and optional SHA256 checksum. Log cache corruption and rebuild automatically.

**[MEDIUM] Missing Error Handling for Empty Embeddings**
- Location: `scripts/cto_context_lib.py` - `search_semantic()`
- Issue: If Ollama returns empty embedding list, cosine similarity will fail with division by zero.
- Evidence: `def cosine_similarity(a: List[float], b: List[float]) -> float:` - no empty list check
- Recommendation: Add `if not a or not b: return 0.0` at start of function. Handle zero-norm vectors.

**[LOW] No Rate Limiting on Ollama API Calls**
- Location: `scripts/cto_context_lib.py` - `embed_text()`
- Issue: Rapid successive calls could overwhelm local Ollama instance.
- Evidence: No rate limiting or backoff mechanism in embedding code
- Recommendation: Add exponential backoff for failed Ollama calls. Implement request queue for batch operations.

**[LOW] JSON Output Exposes Internal Paths**
- Location: `scripts/cto_context_search.py` - JSON output
- Issue: JSON output includes full file paths that could reveal internal directory structure.
- Evidence: `output = {"semantic": [m.to_dict() for m in semantic_matches], "keyword": [m.to_dict() for m in keyword_matches]}` with `path` field
- Recommendation: Consider sanitizing paths in JSON output or adding `--verbose` flag to control path exposure.

**[LOW] No Cache Size Limit**
- Location: `logs/cache/vault_embeddings.json`
- Issue: Cache grows unbounded as files are added; no pruning of deleted files.
- Evidence: `cache = {}` in refresh logic with no cleanup of stale entries
- Recommendation: Add cache cleanup that removes entries for non-existent files. Set maximum cache size with LRU eviction.

### Top 3 Recommendations

1. **Fix path traversal vulnerability in `backup_before_write()`** (relates to CRITICAL finding) - Add path canonicalization and ROOT boundary validation before any file operations. This is the highest risk as it could allow reading/writing arbitrary files on the system.

2. **Add input validation for `--query` and `--limit`** (relates to CRITICAL and HIGH findings) - Implement maximum length (1000 chars) for queries and maximum limit (100) for results. This prevents DoS attacks through resource exhaustion.

3. **Implement atomic cache writes with integrity verification** (relates to HIGH and MEDIUM findings) - Use temp file + rename pattern for cache writes, add JSON schema validation on load, and implement checksum verification. This prevents cache corruption from causing silent failures.

### What I Might Be Wrong About

- **Path traversal severity**: I rated this CRITICAL assuming the backup function is called with user-controlled input. If `backup_before_write()` is only called internally with hardcoded vault paths, the risk is lower. However, the function signature accepts arbitrary `str | Path` which suggests it could be exposed.

- **Ollama security assumptions**: I assumed Ollama running on localhost is secure. If Ollama is misconfigured to accept external connections or if the system is compromised, the embedding API could be abused. I lack context on Ollama's network configuration.

- **Cache corruption likelihood**: I rated cache integrity as MEDIUM assuming corruption is rare. If the system runs on unstable hardware or has frequent crashes, cache corruption could be more common and the impact higher. I don't have data on the deployment environment's reliability.

  - **Automation:** N/A (Requires requirement change).

**[LOW] Vague "Seamlessly" UX Definition**
- Location: Success Metrics section
- Issue: "User can toggle between keyword and semantic seamlessly" is subjective and does not define error handling or latency expectations during the toggle.
- Evidence: "User can toggle between keyword and semantic seamlessly"
- Recommendation: Define specific error handling and latency constraints.
  - **Given** Ollama is running
  - **When** user runs search with `--semantic`

  - **And** no error messages appear in stderr.

1. **Replace subjective relevance metrics with ground-truth test cases** (Relates to Finding 1): Define a fixed set of 5 queries with expected file matches and minimum similarity scores to ensure deterministic verification of semantic search quality.
2. **Quantify "typical delta" for performance gates** (Relates to Finding 2): Specify exact file counts (e.g., "5 modified files") and hardware assumptions (e.g., "SSD, 8GB RAM") to make the <5 second cache refresh target testable.

If this is the case, the metric needs a specific calculation method (e.g., "Sum of token counts of top 5 results").
- **Assumption about hardware constraints:** I rated the performance finding as HIGH severity assuming standard hardware. If the deployment environment is strictly defined (e.g., "AWS t3.medium"), the "typical delta" ambiguity might be less critical than I rated, as the environment is fixed.
- **Assumption about Ollama availability testing:** I assumed killing the Ollama process is sufficient for testing fallback. If the system uses a service wrapper or Docker container that auto-restarts Ollama, the "Ollama unavailable" test might fail to trigger the fallback logic as expected, requiring a network-level mock instead of a process kill.

## Dependency & Integration Risk Mapper (mistral-large)

### Findings

**[CRITICAL] Undefined Embedding Model Version and Stability**
- Location: Phase 1 (Embedding Infrastructure), Ollama API section
- Issue: The spec assumes the use of `nomic-embed-text` but does not specify a version or commit hash. Ollama models are mutable and can be updated without notice, leading to embedding drift or breaking changes.
- Evidence: `"model": "nomic-embed-text"` (Ollama API example) and `"nomic-embed-text model"` (Prerequisites table) are referenced without version pinning.
- Recommendation: Pin the model version in the code (e.g., `nomic-embed-text:latest` → `nomic-embed-text:v1.5.0`) and document the exact model commit hash used during validation. Add a warning in the PRD that future model updates may require re-validation.

---

**[CRITICAL] No Contract for Ollama API Response Shape**
- Location: Ollama API section
- Issue: The spec assumes the Ollama `/api/embeddings` endpoint returns a 768-dimensional vector, but this is not guaranteed. The API contract (response shape, error codes, rate limits) is not defined in the PRD, making the integration unverifiable.
- Evidence: `"embedding": [0.023, -0.145, ...]  // 768 floats` (Ollama API response example) is treated as a fixed contract without validation.
- Recommendation: Define the expected response schema in the PRD, including:
  - `embedding` array length (768)
  - `model` field (must match requested model)
  - Error response shape (e.g., `{"error": "model not found"}`)
  Add runtime validation in `embed_text()` to reject malformed responses.

---

**[HIGH] No Fallback for Embedding Failures Beyond Keyword Search**
- Location: Error Handling table
- Issue: The spec defines fallback to keyword search if Ollama is unavailable, but does not address partial failures (e.g., some files fail to embed, or the model returns invalid vectors). This could silently degrade search quality.
- Evidence: `"Embedding fails | Skip file, continue with others"` (Error Handling table) implies files are dropped without user notification.
- Recommendation: Add a hybrid mode where failed embeddings trigger a warning and include the file in keyword results. Log skipped files to `stderr` with a clear message (e.g., `Warning: Failed to embed 3/99 files. Falling back to keyword for these.`).

---

**[HIGH] Cache Invalidation Relies on mtime Only**
- Location: Phase 3 (Incremental Refresh)
- Issue: The cache invalidation strategy uses file modification time (`mtime`) but does not account for:
  - File content changes without `mtime` updates (e.g., `git checkout`).
  - Changes to embedding logic (e.g., model version updates, text preprocessing changes).
- Evidence: `"Compare mtimes, return files needing re-embed"` (Phase 3) and `get_stale_files()` implementation.
- Recommendation: Add a `version` field to the cache metadata (e.g., `"version": "1.0"`) and invalidate the cache if the embedding logic changes. Include a `--force-refresh` flag to bypass `mtime` checks.

---

**[HIGH] No Rate Limiting or Throttling for Ollama API**
- Location: Embedding Infrastructure
- Issue: The spec does not address rate limits for the Ollama API. Embedding 99 files in a single request could overwhelm the local server or trigger rate limits, leading to failed embeddings.
- Evidence: No mention of rate limits in the Ollama API section or error handling table.
- Recommendation: Add a configurable delay between embedding requests (e.g., 100ms) and batch files into groups of 10-20. Document the assumed rate limit (e.g., "Ollama handles 10 concurrent requests per second").

---

**[MEDIUM] Embedding Target Omits Critical Context**
- Location: Embedding Target section
- Issue: The embedding concatenates only the title, thesis, and tags, omitting the body of the learning. This may reduce recall for queries targeting deeper content (e.g., "AST parsing edge cases" won’t match a learning where the term appears in the body but not the title/thesis).
- Evidence: `"title + ' | ' + thesis + ' | ' + tags"` (Embedding Target section).
- Recommendation: Include the first 500 characters of the body in the embedding target. Document the trade-off (larger cache size vs. improved recall).

---

**[MEDIUM] No Validation of Embedding Quality**
- Location: Evaluation Criteria
- Issue: The spec evaluates semantic search by comparing it to keyword search but does not validate that the embeddings themselves are meaningful (e.g., "agent memory" should be closer to "memory systems" than to "Ollama setup").
- Evidence: `"Find learnings keyword misses"` (Evaluation Criteria) focuses on recall but not semantic coherence.
- Recommendation: Add a validation step to the PRD: manually inspect the top 5 results for 10 semantically related queries (e.g., "RAG performance" vs. "why does retrieval fail") and document the expected vs. actual matches.

---

**[MEDIUM] Cache File Location Not Version-Controlled**
- Location: Phase 3 (Incremental Refresh)
- Issue: The cache file (`logs/cache/vault_embeddings.json`) is stored in a directory that may be git-ignored (`logs/`). This could lead to cache drift across environments or team members.
- Evidence: `"Cache location: logs/cache/vault_embeddings.json"` (Verification section).
- Recommendation: Move the cache to a version-controlled directory (e.g., `scripts/cache/`) or document that the cache is environment-specific and must be regenerated on checkout.

---

**[LOW] No Handling for Large Files or Timeouts**
- Location: Embedding Infrastructure
- Issue: The spec does not address timeouts or large files (e.g., a 10MB learning file). Ollama may fail to embed large inputs, leading to silent failures.
- Evidence: No mention of file size limits or timeout handling in the Ollama API section.
- Recommendation: Add a timeout (e.g., 30 seconds) to `embed_text()` and skip files larger than 1MB. Log a warning for skipped files.

---

**[LOW] No Documentation for Cache Schema**
- Location: Phase 3 (Incremental Refresh)
- Issue: The cache schema (e.g., `{"embedding": [...], "mtime": ...}`) is not documented in the PRD, making it harder to debug or extend.
- Evidence: `"Returns {filepath: {"embedding": [...], "mtime": ...}}"` (Phase 1) is the only schema reference.
- Recommendation: Add a "Cache Schema" section to the PRD with a JSON example and field descriptions (e.g., `mtime: Unix timestamp of file modification`).

---

### Top 3 Recommendations

1. **Pin the Ollama model version and validate the API contract** (CRITICAL findings). Without this, the integration is unverifiable and may break silently.
2. **Add hybrid fallback for partial embedding failures** (HIGH finding). Skipping files without notification could degrade search quality without the user’s knowledge.
3. **Improve cache invalidation with versioning and force-refresh** (HIGH finding). Relying solely on `mtime` risks stale or corrupted caches.

---

### What I Might Be Wrong About

- **Assumption that Ollama’s `/api/embeddings` endpoint is stable**: If Ollama’s API is versioned and backward-compatible, the risk of embedding drift may be lower than assumed. However, the PRD does not confirm this.
- **Impact of omitting the learning body from embeddings**: The trade-off between cache size and recall may be acceptable if the title/thesis/tags are sufficiently representative. This depends on the actual content of the learnings.
- **Severity of rate limiting**: If Ollama’s local server is designed for high throughput, rate limiting may not be necessary. However, the PRD does not provide evidence of Ollama’s performance characteristics.
