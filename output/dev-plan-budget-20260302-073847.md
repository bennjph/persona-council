## Deployment Risk Assessor (deepseek-v3.2)

### Findings

**[CRITICAL] No Rollback Path for Cache Corruption or Schema Changes**
- Location: Phase 1: Embedding Infrastructure, Implementation Details: Embedding Target
- Issue: The plan introduces a new data store (`logs/cache/vault_embeddings.json`) with a specific schema but provides no way to revert to a previous, known-good state if the cache becomes corrupted or the embedding schema changes.
- Evidence: "Cache management — `logs/cache/vault_embeddings.json`" and "Cache corrupt | Rebuild from scratch". The only recovery from corruption is a full rebuild, which is not a rollback and could be time-consuming or impossible if the source data has changed.
- Recommendation: Implement a versioned backup system for the cache. Before any cache write (full refresh or incremental update), create a timestamped copy of the existing cache file in a `logs/cache/backups/` directory, retaining the last N backups. Provide a `--restore-cache <timestamp>` CLI flag to revert.

**[HIGH] Blast Radius of Cache Staleness Affects All Semantic Search Users**
- Location: Phase 3: Incremental Refresh, Implementation Details: Error Handling
- Issue: If the incremental refresh logic fails silently (e.g., `get_stale_files` has a bug, file mtime comparison is wrong), users will receive stale or missing semantic search results without warning, degrading the service for all users simultaneously.
- Evidence: "Smart caching — Only re-embed files modified since last cache" and "Embedding fails | Skip file, continue with others". A failure in the refresh mechanism is not isolated; it pollutes the shared cache.
- Recommendation: Add a cache health check and audit log. The `search_semantic` function should log a warning if the cache's aggregate "mtime" is older than a threshold (e.g., 7 days). Implement a dry-run mode for `--refresh-cache` that reports which files would be re-embedded and why.

**[MEDIUM] No Feature Flag for Semantic Search Integration**
- Location: Phase 2: Semantic Search Implementation, Execution Sequence
- Issue: The `--semantic` flag enables a major new search path, but there is no mechanism to disable it at runtime if it causes performance issues or incorrect results, requiring a code redeploy to revert to keyword-only.
- Evidence: "Add `--semantic` flag" and "If `--semantic` and Ollama available: ... Else: fall back to keyword search". The control is a CLI flag, not a runtime feature flag that can be toggled without modifying invocation scripts or redeploying code.
- Recommendation: Implement a feature flag system (e.g., a simple `config.json` file or environment variable) that controls whether the semantic search path is *advertised* or *active*. The CLI should respect this flag, hiding or disabling the `--semantic` option entirely if the feature is turned off.

**[MEDIUM] Deployment Sequence Lacks Validation of External Dependency Health**
- Location: Prerequisites, Execution Sequence
- Issue: The plan's execution sequence begins coding immediately if Ollama is "ready," but there is no validation step to ensure the Ollama service remains stable and responsive *throughout* the deployment, leading to potential mid-deploy failures.
- Evidence: "Week 1 (if Ollama ready)" and "Test: Verify Ollama connectivity". A single health check at the start does not guarantee the service will be available during cache population, which is a multi-file, sequential operation.
- Recommendation: Add a pre-flight check that validates Ollama stability: perform a small batch of embedding requests (e.g., 5 files) and measure latency/error rate. If stability is poor, abort the deployment with a clear message. Wrap the embedding loop with a circuit breaker pattern to stop and alert if consecutive requests fail.

**[LOW] Data Migration Risk from Embedding Model Changes**
- Location: Architecture, Implementation Details: Ollama API
- Issue: If the embedding model is changed (e.g., from `nomic-embed-text` to another), all existing cache entries become invalid, requiring a full, manual rebuild. The plan does not account for this versioning.
- Evidence: `def embed_text(text: str, model: str = "nomic-embed-text")` and the cache schema `{filepath: {"embedding": [...], "mtime": ...}}`. The cache does not store the model version used to generate the embeddings.
- Recommendation: Include a `model_version` or `model_name` field in the cache metadata (`vault_embeddings.json`). The `load_embedding_cache` function should validate that the cached model matches the current configured model and invalidate the cache (or trigger a full refresh) if they differ.

### Top 3 Recommendations

1. **Implement versioned backups for the embedding cache** to enable instant rollback from corruption or bad embeddings. This addresses the CRITICAL finding of no rollback path.
2. **Add a runtime feature flag to control semantic search availability**, not just a CLI flag, to allow disabling the new functionality without code changes. This addresses the MEDIUM finding on lack of operational control.
3. **Introduce a stability pre-check and circuit breaker for Ollama calls** during cache population to prevent a partial, corrupted cache state from being saved. This addresses the HIGH finding on blast radius from cache staleness.

### What I Might Be Wrong About

- **Assumption that `logs/cache/vault_embeddings.json` is a critical production data store.** It might be considered a disposable, rebuildable cache. If so, the CRITICAL finding is overstated, and a simple "delete and rebuild" is an acceptable rollback, though it carries user-facing latency cost.
- **Area where I lack context about user invocation patterns.** I assume `cto_context_search.py` is called by automated agents or frequent human users. If it's only used sporadically by a single user, the blast radius of cache issues (HIGH finding) is much smaller, reducing the severity.
- **Finding that the lack of a model version in the cache is a MEDIUM risk.** This might be LOW if changing the embedding model is a rare, planned event that would naturally coincide with a full cache refresh, making manual intervention acceptable.

If the developer considers Ollama an "embedded-style" local dependency, they might view my request for correlation IDs as over-engineering. However, in production-grade SRE, there is no such thing as a "reliable" local network call.

## Coupling Detector (qwen-3.5-27b)

### Findings

**[HIGH] Dynamic Import Creates Hidden Dependency**
- Location: `scripts/cto_context_search.py:35-41`
- Issue: The `--refresh-cache` path imports functions dynamically inside the function body, hiding dependencies from static analysis and making the module harder to test in isolation.
- Evidence: `from cto_context_lib import (load_embedding_cache, save_embedding_cache, _extract_learning_text, embed_text, ROOT)` appears inside the `main()` function after argument parsing, not at module level.
- Recommendation: Move all imports to module top level. This exposes the full dependency surface and enables static analysis tools to catch missing functions.

**[HIGH] Shared Cache File Creates Implicit Contract**
- Location: `logs/cache/vault_embeddings.json` (referenced by both `cto_context_lib.py` and `cto_context_search.py`)
- Issue: The embedding cache is a shared mutable state file that couples multiple modules through an implicit JSON schema contract. Any format change breaks all consumers without compile-time detection.
- Evidence: Cache structure `{filepath: {"embedding": [...], "mtime": ..., "title": ...}}` is documented in the plan but not enforced by any schema or type system.
- Recommendation: Define a dataclass or TypedDict for the cache entry structure and validate on load/save. Add version field to cache file for future schema migrations.

**[MEDIUM] Ollama Availability Check Creates Temporal Coupling**
- Location: `scripts/cto_context_search.py:58-62`
- Issue: The `ollama_available()` check happens before search, but Ollama could fail between the check and actual embedding calls, creating a race condition with undefined behavior.
- Evidence: `if not ollama_available(): print("Warning: Ollama not available. Falling back to keyword search.")` followed by `semantic_matches = search_semantic(...)` without wrapping the actual call in try/except.
- Recommendation: Wrap `search_semantic()` call in try/except for connection errors. Log specific error type (connection refused vs timeout vs model not found) for better diagnostics.

**[MEDIUM] ROOT Path Coupling to Filesystem Layout**
- Location: `scripts/cto_context_lib.py:14`
- Issue: `ROOT = Path(__file__).resolve().parents[1]` hardcodes the assumption that `cto_context_lib.py` is always in `scripts/` directory. Moving the file breaks all path computations.
- Evidence: `ROOT = Path(__file__).resolve().parents[1]` with no configuration override or environment variable support.
- Recommendation: Accept ROOT as a parameter or environment variable (`CTO_VAULT_ROOT`). Default to current behavior for backward compatibility but allow override for testing or deployment.

**[MEDIUM] SOURCE_WEIGHTS Shared Configuration Affects Both Search Paths**
- Location: `scripts/cto_context_lib.py:33-40`
- Issue: `SOURCE_WEIGHTS` dictionary is used by keyword search but the plan doesn't clarify if semantic search should respect the same weights. This creates ambiguity about whether changing weights affects both search modes.
- Evidence: `SOURCE_WEIGHTS = {"02-validated/patterns/context-packs": 6, "fundamentals": 5, ...}` is module-level constant used in `_iter_markdown_files()` which is called by `search_context()`.
- Recommendation: Document explicitly whether semantic search uses `SOURCE_WEIGHTS`. If yes, add a parameter to `search_semantic()` to allow different weighting. If no, extract keyword-specific weights to avoid confusion.

**[LOW] Backup Directory Not Integrated with Semantic Search**
- Location: `scripts/cto_context_lib.py:17-18` vs Plan 18 "Related" section
- Issue: `BACKUP_DIR` and `backup_before_write()` exist in the library but the plan doesn't specify when/why they should be used for cache writes. The "Shadow restore points" pattern from the learning document isn't applied to cache management.
- Evidence: `BACKUP_DIR = ROOT / "logs" / "backups"` is defined but `save_embedding_cache()` (mentioned in plan) doesn't show backup logic in the visible code.
- Recommendation: Either document that cache doesn't need backups (low-risk data) or add `backup_before_write()` call to `save_embedding_cache()` for consistency with the "shadow restore points" pattern.

**[LOW] Cache Size Not Monitored**
- Location: Plan 18 "Notes" section
- Issue: The plan notes "Monitor cache size; 100 learnings × 768 dims ≈ 300KB uncompressed" but there's no implementation for monitoring or alerting when cache exceeds expected bounds.
- Evidence: "Monitor cache size" is listed as a note with no corresponding action item or implementation detail.
- Recommendation: Add cache size check in `save_embedding_cache()` that logs a warning if file exceeds 10MB threshold. Add `--cache-stats` flag to display current cache size and entry count.

### Top 3 Recommendations

1. **Move dynamic imports to module level** (relates to HIGH: Dynamic Import Creates Hidden Dependency) — This exposes the full dependency surface and enables static analysis. The 5-line change prevents future debugging nightmares when functions are renamed or removed.

2. **Add schema validation to cache file** (relates to HIGH: Shared Cache File Creates Implicit Contract) — Define a TypedDict for cache entries and validate on load. This prevents silent corruption when the format changes and makes the contract explicit.

3. **Wrap semantic search in try/except** (relates to MEDIUM: Ollama Availability Check Creates Temporal Coupling) — The availability check is a race condition. Wrap the actual embedding calls to handle connection failures gracefully and log specific error types.

### What I Might Be Wrong About

- **The dynamic import might be intentional for lazy loading** — If `embed_text()` has heavy dependencies (e.g., numpy), the dynamic import could be a performance optimization. However, the code shows `numpy` is already imported at module level in the plan's prerequisites, so this is unlikely.

- **The cache file might be read-only for search operations** — If `search_semantic()` only reads the cache and only `--refresh-cache` writes it, the coupling is less severe. But the plan mentions "incremental refresh" which implies writes during search, creating the coupling I flagged.

- **The ROOT path coupling might be acceptable for this project's scope** — If the vault is never deployed outside the current filesystem layout, the hardcoded path is fine. However, this prevents testing the library in isolation and makes CI/CD harder if the project grows.
