## Requirements Archaeologist (gpt-5.1-codex-mini)

### Findings

**[HIGH] Missing operations monitoring for Ollama availability**
- Location: Risks & Mitigations / Evaluation Criteria
- Issue: The plan relies on local Ollama but only specifies a warning/fallback in the CLI; it never requires instrumentation, alerting, or ownership to ensure Ollama is kept running once availability degrades, leaving the semantic path silently broken until someone notices.
- Evidence: “Warning: Ollama not available. Falling back to keyword search.” (Implementation Complete) and “Fallback reliability — Stop Ollama mid-search | Graceful keyword fallback” (Evaluation Criteria) describe what happens when Ollama fails but not who is responsible for restoring service or how repeated fallbacks are detected.
- Recommendation: Define an operational requirement that tracks Ollama health (e.g., log/error counters, alert if fallback occurs >N times/hour) and assign an owner/response playbook so the dependency isn’t silently lost in production.

**[HIGH] Security stakeholder needs for sensitive content embedding are missing**
- Location: Implementation Details / Action Items
- Issue: Semantic indexing is applied to all learnings without any requirement to exclude or redact sensitive transcripts or personal data, so embedding can inadvertently expose PII through similarity search and violate data governance policies.
- Evidence: “Embed these fields from each learning: Title … Thesis … Tags … Concatenate: …” (Implementation Details) but nowhere does the plan require filtering by sensitivity, honor `include-transcripts`, or let a security reviewer opt files out.
- Recommendation: Add a requirement that only approved directories or metadata-labeled files are fed into the embedding cache (e.g., skip transcripts unless explicitly allowed), document the review process, and enforce it in the cache-refresh logic.

**[MEDIUM] Cache TTL is mentioned but not implemented**
- Location: Phase 3 / Action Items
- Issue: The plan claims “Cache TTL — Configurable (default: 24 hours)” but never specifies where or how TTL is enforced (no requirement to invalidate/recompute stale cache entries based on elapsed time), so embeddings may stay stale indefinitely despite the stated goal.
- Evidence: “Cache TTL — Configurable (default: 24 hours)” (Phase 3) is the sole mention; neither the “Search flow” nor “Implementation Complete” sections describe what component checks this TTL before using cached vectors.
- Recommendation: Require the semantic search workflow to compare each cache entry’s timestamp against the TTL and schedule or trigger re-embedding when expired, so the stated TTL becomes actionable and prevents outdated results.

**[MEDIUM] Assumes embedding dimension is fixed without verification**
- Location: Implementation Details
- Issue: The plan embeds using nomic-embed-text and assumes the returned vector is always 768 floats, but it does not require a validation step, so a changed/OpenAI response shape would silently break similarity math or corrupt the cache.
- Evidence: “Response: { 'embedding': [0.023, -0.145, ...] // 768 floats }” (Ollama API) indicates the expectation but no subsequent validation or safeguard is described.
- Recommendation: Specify a requirement to assert the received vector length (and type) before caching/using it, log mismatches, and fall back to keyword search when validation fails, preventing silent corruption.

**[LOW] Cache cleanup for deleted or renamed files is undefined**
- Location: Phase 1 / Action Items
- Issue: The plan defines `get_stale_files()` to re-embed modified files but never requires detecting or removing entries for files that were deleted or renamed, leaving orphaned cache entries that will grow unbounded and point to missing paths.
- Evidence: “get_stale_files() — Compare mtimes, return files needing re-embed” (Phase 1) limits the logic to modified files, not removals.
- Recommendation: Add a requirement for cache synchronization that removes entries whose source files no longer exist or have been renamed, keeping the cache accurate and preventing dead references.

### Top 3 Recommendations

1. Implement actionable monitoring/alerting for Ollama availability (Finding: Missing operations monitoring for Ollama availability).
2. Enforce the declared cache TTL by invalidating/re-embedding entries older than the configured window (Finding: Cache TTL is mentioned but not implemented).
3. Introduce opt-in filtering/redaction for sensitive transcripts before embedding to satisfy security stakeholders (Finding: Security stakeholder needs for sensitive content embedding are missing).

### What I Might Be Wrong About

- I assumed the plan is intended for a regulated environment where embedding transcripts is sensitive; if the vault only contains public technical notes, the security gap is less critical.
- I lack context about how often Ollama actually fails in the current setup; if it’s extremely reliable and monitored elsewhere, the lack of local alerts may not be addressed in this plan.
- The TTL gap might already be implemented in code paths not shown in this plan text; if the cached metadata includes expiry checks, the severity I assigned could be overstated.

## Spec Prosecutor (deepseek-v3.2)

### Findings

**[CRITICAL] Ambiguous "Embedding Target" Definition**
- Location: Implementation Details
- Issue: The spec defines what to embed inconsistently, causing different implementations to embed different text, breaking similarity comparisons.
- Evidence: "Embed these fields from each learning: - **Title** (first `# ` heading) - **Thesis line** (first paragraph after title) - **Tags** (if present). Concatenate: `title + " | " + thesis + " | " + tags`"
- Recommendation: Define the exact extraction algorithm: 1) Title is the first line starting with '# ' stripped of '# '. 2) Thesis line is the first non-empty, non-title line after the title. 3) Tags are all words following `**Tags**:` on the same line. Provide an example.

**[HIGH] Contradiction on Cache Refresh Trigger**
- Location: Phase 3: Incremental Refresh vs. `get_stale_files` function
- Issue: The plan says to refresh based on a "TTL (default: 24 hours)" but the implemented function `get_stale_files` compares file modification times (`mtime`), not a time-to-live.
- Evidence: Phase 3 states: "**Cache TTL** — Configurable (default: 24 hours)". The verification section states: "Smart refresh: only re-embeds modified files" and references `get_stale_files()` which compares `mtime`.
- Recommendation: Choose one strategy: either TTL-based refresh or mtime-based refresh. Update the spec and function name/behavior to be consistent. If TTL is desired, the cache must store a timestamp of when the embedding was created.

**[HIGH] Undefined Behavior for Missing "Thesis line" or "Tags"**
- Location: Implementation Details
- Issue: The embedding target algorithm does not define what to do if a learning file lacks a "thesis line" paragraph or a "Tags" field, leading to malformed concatenated strings or errors.
- Evidence: "Embed these fields from each learning: - **Title** ... - **Thesis line** ... - **Tags** (if present). Concatenate: `title + " | " + thesis + " | " + tags`"
- Recommendation: Explicitly define fallback values: If "thesis line" is absent, use an empty string. If "Tags" is absent, omit the " | " separator and tags portion entirely. The concatenation logic must handle these cases.

**[MEDIUM] Contradictory Search Output Formats**
- Location: Phase 2: Semantic Search Implementation vs. Verification Test Results
- Issue: The spec prescribes a markdown table output format, but the implemented output (shown in verification) uses a different, non-tabular format.
- Evidence: Phase 2 states: "**Display format** (add to existing) ```markdown ## Semantic Results (Top 5) | Score | File | Excerpt | |-------|------|---------|```". Verification shows: "`[Score: 0.717] knowledge/learnings/...`".
- Recommendation: Standardize the output format. Decide if the new semantic results should be integrated into the existing keyword result format (as shown) or use a separate table. Update the spec to match the chosen implementation.

**[MEDIUM] Ambiguous "Ollama available" Health Check**
- Location: Error Handling & `ollama_available()` function
- Issue: The condition for falling back to keyword search is ambiguous. Is Ollama "not available" if the daemon isn't running, if the model isn't pulled, if the `/api/embeddings` endpoint times out, or if it returns an error?
- Evidence: Error Handling table: "Ollama not running | Log warning, fallback to keyword". The function `ollama_available()` is not defined in the spec.
- Recommendation: Define the exact health check: e.g., a GET request to `http://localhost:11434/api/tags` with a 2-second timeout. Specify that any failure (connection error, timeout, non-200 response) triggers the fallback.

**[LOW] Undefined Boundary for "Top-5" Results**
- Location: Phase 2: Semantic Search Implementation
- Issue: The spec says "Return top-5 by cosine similarity + scores" but does not define behavior when fewer than 5 files exist or when similarity scores are below a meaningful threshold (e.g., all scores < 0.1).
- Evidence: "Return top-5 by cosine similarity + scores"
- Recommendation: Specify: Return up to `limit` (configurable, default 5) results, sorted descending by score. If no files have a similarity score > 0 (or a configurable minimum threshold), return an empty list.

**[LOW] Missing State Transition for Cache Corruption**
- Location: Error Handling
- Issue: The spec says "Cache corrupt | Rebuild from scratch" but does not define how corruption is detected or the sequence of operations (e.g., delete corrupt file, re-embed all).
- Evidence: Error Handling table: "Cache corrupt | Rebuild from scratch"
- Recommendation: Define detection (e.g., JSON decode fails, missing required keys) and the automatic recovery procedure: log error, delete `vault_embeddings.json`, call the refresh-cache logic.

### Top 3 Recommendations

1. **[CRITICAL Fix]** Precisely define the text extraction algorithm for embeddings, including handling missing fields, to ensure consistent vector generation across runs and implementations.
2. **[HIGH Fix]** Resolve the contradiction between TTL-based and mtime-based cache refresh by choosing one strategy and updating all spec sections and function names/logic accordingly.
3. **[HIGH Fix]** Define complete behavior for edge cases in the embedding target (missing thesis, missing tags) to prevent runtime errors or malformed data.

- **Assumption about `get_stale_files` implementation**: I assumed `get_stale_files` only uses `mtime` because the verification text says "modified files." It's possible the function also implements a TTL check internally. If so, the contradiction is only in the spec text, not the code.
- **Context of output format**: The spec's "Display format" might have been an initial proposal that was intentionally changed during implementation to better integrate with the existing keyword output style. The "contradiction" might be an accepted evolution, not a bug.
- **Severity of cache TTL vs. mtime**: I rated the cache refresh contradiction as HIGH. However, if the primary use case is a personal vault where files are edited manually, mtime-based refresh is more intuitive and correct than TTL. The mention of "TTL" might be a vestigial requirement from an earlier draft, making this a MEDIUM documentation cleanup issue.

## Edge Case Hunter (qwen-3.5-27b)

### Findings

**[CRITICAL] Non-Atomic Cache Write Risk**
- Location: Plan 18 — Phase 1 Action Item 3
- Issue: The plan specifies "Atomic write to cache file" for `save_embedding_cache`, but the implementation details are hidden in `cto_context_lib.py`, creating a risk of partial JSON writes corrupting the cache on interruption.
- Evidence: Plan states: `def save_embedding_cache(cache: Dict): # Atomic write to cache file` without showing implementation; `scripts/cto_context_search.py` calls `save_embedding_cache(cache)` directly.
- Recommendation: Implement `save_embedding_cache` using `tempfile.mkstemp` followed by `os.replace` to guarantee atomicity, and wrap in a `try/finally` block to clean up temp files on failure.

**[HIGH] Cache File Race Condition**
- Location: Plan 18 — Phase 3 Incremental Refresh
- Issue: Concurrent execution of `--refresh-cache` and `--semantic` search can cause `load_embedding_cache` to read a partially written or locked file, leading to JSON decode errors or inconsistent state.
- Evidence: Plan states: "Cache location: `logs/cache/vault_embeddings.json`" and "Smart refresh: only re-embeds modified files" without specifying file locking mechanisms.
- Recommendation: Implement file locking (e.g., `fcntl.flock` on Unix or `msvcrt.locking` on Windows) around both read and write operations on `vault_embeddings.json`.

**[HIGH] Ollama Network Timeout Missing**
- Location: Plan 18 — Error Handling Table
- Issue: The `ollama_available` and `embed_text` functions lack explicit timeout configurations, allowing the script to hang indefinitely if the Ollama service is unresponsive but not dead.
- Evidence: Plan states: "Ollama not running | Log warning, fallback to keyword" but does not specify timeout duration for the HTTP request to `localhost:11434`.
- Recommendation: Add a `timeout` parameter (e.g., 5 seconds) to all `urllib.request` or `requests` calls interacting with the Ollama API, catching `socket.timeout` to trigger fallback.

**[MEDIUM] Query Input Length Unbounded**
- Location: `scripts/cto_context_search.py` — `build_parser`
- Issue: The `--query` argument accepts strings of arbitrary length, enabling denial-of-service via memory exhaustion during tokenization or embedding generation.
- Evidence: Code shows: `parser.add_argument("--query", required=True, help="Query string.")` with no `type` constraint or length validation.
- Recommendation: Add a `type=str` validator or post-parse check to limit `--query` to a maximum length (e.g., 500 characters) and reject oversized inputs with a clear error message.

**[MEDIUM] Mtime-Based Staleness Fragility**
- Location: Plan 18 — Phase 3 Action Item 1
- Issue: Relying solely on file modification time (`mtime`) for cache invalidation is vulnerable to clock skew, `touch` commands, or system restore operations that invalidate the cache incorrectly.
- Evidence: Plan states: "Compare mtimes, return files needing re-embed" and `scripts/cto_context_lib.py` shows: `"mtime": filepath.stat().st_mtime`.
- Recommendation: Supplement `mtime` with a content hash (e.g., SHA-256 of the first 1KB) to verify actual content changes before re-embedding.

**[MEDIUM] Path Traversal in File Iteration**
- Location: `scripts/cto_context_lib.py` — `_iter_markdown_files`
- Issue: The function iterates `folder.rglob("*.md")` without validating that the resolved path remains within the intended `ROOT` directory, potentially exposing files outside the vault if symlinks are manipulated.
- Evidence: Code shows: `for path in folder.rglob("*.md"):` without `path.resolve().is_relative_to(ROOT)` check.
- Recommendation: Add a check `if not path.resolve().is_relative_to(ROOT): continue` inside the iteration loop to prevent directory traversal via symlinks.

### Top 3 Recommendations

1. **Implement Atomic Cache Writes** (Relates to CRITICAL finding): Refactor `save_embedding_cache` to write to a temporary file in the same directory and use `os.replace` to swap it into place, ensuring the cache file is never in a partial state.
2. **Add File Locking to Cache Operations** (Relates to HIGH finding): Wrap all reads and writes to `logs/cache/vault_embeddings.json` with an exclusive lock to prevent race conditions between search and refresh operations.
3. **Enforce Network Timeouts for Ollama** (Relates to HIGH finding): Configure a strict timeout (e.g., 5s) on all HTTP requests to the Ollama API to prevent the CLI from hanging indefinitely during service degradation.

### What I Might Be Wrong About

- **Atomic Write Implementation**: I assumed the "Atomic write" claim in the Plan was unimplemented or risky because the code was truncated. If the actual `cto_context_lib.py` implementation uses `tempfile` and `os.replace` (standard Python pattern), the CRITICAL severity is overstated.
- **Ollama Timeout Behavior**: I assumed the underlying HTTP client (likely `urllib` or `requests`) defaults to no timeout or a very long one. If the Ollama client library used internally has aggressive default timeouts, the hang risk is lower than assessed.
- **Mtime Reliability**: I rated `mtime` fragility as MEDIUM. If this system is strictly single-user on a local machine with no external clock synchronization issues, `mtime` might be sufficiently reliable for the intended "agent-level undo" use case, making the hash recommendation lower priority.

## Dependency & Integration Risk Mapper (mistral-large)

- Location: Phase 1: Embedding Infrastructure, Ollama API
- Issue: The spec assumes the use of `nomic-embed-text` but does not specify a required version or stability level. Ollama models are community-driven and can be updated or deprecated without notice, risking silent degradation or failure of semantic search.

**[CRITICAL] No Fallback Behavior for Embedding Cache Corruption**
- Location: Phase 3: Incremental Refresh, Cache TTL
- Issue: The spec states that if the cache is corrupt, it should be rebuilt from scratch, but it does not define what happens during the rebuild process. If the cache is corrupted mid-search, the system may fail to return any results, breaking user-facing functionality.
- Evidence: `"Cache corrupt | Rebuild from scratch"` in the Error Handling table; no interim behavior defined.
- Recommendation: Implement a two-phase cache recovery: (1) attempt to load the last known good cache from a backup (e.g., `vault_embeddings.json.bak`), and (2) if that fails, fall back to keyword search until the cache is rebuilt. Log the incident for observability.

**[HIGH] No SLA or Latency Requirements for Ollama Embedding Service**
- Location: Phase 2: Semantic Search Implementation, Search Flow
