## Architecture Stress Tester (codex)

### Findings

**[CRITICAL] Cache Schema Can Serve Invalid Similarity Scores After Model Changes**
- Location: Action Items > Phase 1: Cache management
- Issue: Cache validity is keyed by file path and mtime only, so changing embedding model/version can silently corrupt ranking quality.
- Evidence: `"Returns {filepath: {"embedding": [...], "mtime": ...}}"` and `def embed_text(text: str, model: str = "nomic-embed-text")`.
- Recommendation: Add `model_name`, `model_digest`, `embedding_dim`, and `content_hash` to each cache record; hard-invalidate entries when any of those fields differ.

**[HIGH] Ollama on Localhost Is a Runtime Single Point of Failure for Semantic Mode**
- Location: Implementation Details > Ollama API
- Issue: Semantic retrieval depends on one local process and one host endpoint, so service interruption removes semantic capability entirely.
- Evidence: `"POST to localhost:11434/api/embeddings"` and `"Ollama not running | Log warning, fallback to keyword"`.
- Recommendation: Make endpoint configurable (`primary`, `secondary`), add request timeout/circuit breaker, and keep serving from last-good index when embedding service is unavailable.

**[HIGH] Single JSON Cache File Is a Concurrency and Durability Bottleneck**
- Location: Action Items > Phase 1: Cache management
- Issue: A monolithic JSON file will become a write hotspot and race point under concurrent refresh/search executions.
- Evidence: `"Cache management — logs/cache/vault_embeddings.json"` and `"save_embedding_cache(cache: Dict): # Atomic write to cache file"`.
- Recommendation: Replace JSON with SQLite/DuckDB (WAL mode) and row-level upserts keyed by file ID; enforce file locks for refresh jobs.

**[HIGH] Query Path Performs Index Maintenance, Creating Synchronous Tail-Latency Spikes**
- Location: Action Items > Phase 2: Search flow
- Issue: User queries can block on stale-file re-embedding, coupling read latency to indexing workload.
- Evidence: `"If --semantic ... Check cache freshness ... Re-embed stale files ... Embed query ... Return top-5"`.
- Recommendation: Move refresh to a background indexer; query path should read only from a stable snapshot and never wait on embedding generation.

**[HIGH] Brute-Force Top-K Cosine Search Will Degrade at 10x/100x Corpus Size**
- Location: Action Items > Phase 2: Search flow
- Issue: Linear scan ranking across all vectors is acceptable now but will produce rising p95 latency as corpus grows.
- Evidence: `"Return top-5 by cosine similarity + scores"` and `"99 learnings embedded"`.
- Recommendation: Add ANN indexing (HNSW/IVF) with pre-normalized vectors; keep BM25+vector hybrid reranking for quality control.

**[HIGH] Embedding Refresh Uses Per-File Synchronous Calls (N+1 I/O Pattern)**
- Location: Action Items > Phase 1: Embed function / Phase 3: Incremental refresh
- Issue: One HTTP call per file does not scale; refresh throughput will saturate network/process limits first.
- Evidence: `"def embed_text(text: str ...): # POST to localhost:11434/api/embeddings"` and `"Only re-embed files modified since last cache"`.
- Recommendation: Implement bounded parallel workers with backpressure and request batching where supported; cap refresh work per run.

**[MEDIUM] Cache Lifecycle Does Not Specify Deletion/Reconciliation Semantics**
- Location: Action Items > Phase 1: get_stale_files
- Issue: Comparing mtimes alone can leave orphaned cache entries for deleted/renamed files, causing dead paths in results.
- Evidence: `"get_stale_files(cache: Dict) -> List[Path]: # Compare mtimes, return files needing re-embed"`.
- Recommendation: During refresh, reconcile cache keys against live filesystem and remove tombstoned entries before search.

**[HIGH] Completion Status Is Inconsistent With Unmet Acceptance Criteria**
- Location: Header / Success Metrics / Verification
- Issue: The plan is marked complete without proof that declared success gates were met.
- Evidence: `"Status: ✅ Completed 2026-03-01"` and `"- [ ] Semantic search finds ≥3 relevant learnings that keyword search misses"`.
- Recommendation: Gate completion on explicit benchmark receipts (10-query set, p50/p95 latency, hit-quality delta) and fail completion when checkboxes remain open.

**[HIGH] Isolation Boundaries Are Missing; Search and Indexing Form a Distributed Monolith-in-One-Script**
- Location: Architecture
- Issue: Keyword routing, semantic ranking, embedding generation, and cache mutation are coupled in one runtime boundary.
- Evidence: Architecture diagram shows all components inside `"cto_context_search.py"` with internal `"Router"`, `"Semantic Path"`, and `"Embedding Cache (JSON)"`.
- Recommendation: Split into independent components: `indexer` (build/update embeddings), `query service` (read-only retrieval), and `storage` (versioned index); deploy and scale each independently.

### Top 3 Recommendations

1. Implement cache versioning and invalidation by `model_digest + embedding_dim + content_hash` immediately (Findings: **Cache Schema Can Serve Invalid Similarity Scores**, **Single JSON Cache File Is a Concurrency and Durability Bottleneck**).
2. Remove indexing work from the query path and introduce background snapshot-based indexing (Findings: **Query Path Performs Index Maintenance**, **Isolation Boundaries Are Missing**).
3. Replace brute-force scan + per-file synchronous embedding with ANN retrieval and bounded parallel ingestion (Findings: **Brute-Force Top-K Cosine Search**, **Per-File Synchronous Calls**).

### What I Might Be Wrong About

- The system may be intentionally single-user/local-only; if true, concurrency and SPOF findings drop in severity, but model-version invalidation remains critical.
- The implemented code may already include model/version metadata not shown in this plan text; if true, the cache-corruption finding is overstated.
- Corpus growth may be capped near current size; if it stays under a few hundred documents, ANN/indexing urgency is lower and linear scan may remain acceptable.
- A separate scheduler may already run `--refresh-cache` outside the query path; if true, the synchronous-chain finding is less severe.

## Deployment Risk Assessor (codex)

### Findings

**[CRITICAL] Rollback Procedure Is Missing for a Marked-Complete Release**
- Location: Verification; Risks & Mitigations
- Issue: The plan is marked complete without an explicit, tested rollback path for code and cache changes.
- Evidence: "**Status**: ✅ Completed 2026-03-01" and the mitigation list only covers setup/latency/quality risks, not rollback steps.
- Recommendation: Add a `Rollback` section with exact commands and order: revert commit(s), clear or downgrade cache, run smoke query, and verify fallback; require a rollback drill before release sign-off.

**[CRITICAL] Cache Format Has No Versioning or Compatibility Contract**
- Location: Phase 1: Cache management; Error Handling
- Issue: The cache schema is unversioned, so rolling back code can fail or silently misread new cache data.
- Evidence: `Returns {filepath: {"embedding": [...], "mtime": ...}}` and `Cache corrupt | Rebuild from scratch`.
- Recommendation: Add `schema_version` to cache, implement dual-read (`v1`/`v2`) during transition, and include a downgrade path (`v2 -> v1` or forced rebuild) documented in rollback steps.

**[HIGH] Deployment Sequence Allows Dependency Mismatch Between Files**
- Location: Action Items (Phase 1 and Phase 2); Execution Sequence
- Issue: Splitting library and caller refactors across separate days creates a window where one deploy can reference missing interfaces.
- Evidence: `Phase 1 ... scripts/cto_context_lib.py (additions)` and `Phase 2 ... scripts/cto_context_search.py (major refactor)` with `Day 1` then `Day 2-3`.
- Recommendation: Deploy both files as one atomic artifact, or introduce backward-compatible shims so either version of caller/lib works during staggered rollout.

**[HIGH] Blast Radius Is Not Explicitly Mapped for a Shared Search Path**
- Location: Objective; Related
- Issue: A core search utility is changed without identifying dependent workflows, so failure can cascade beyond this feature.
- Evidence: `Upgrade scripts/cto_context_search.py` and `scripts/cto_context_search.py — Target file for upgrade`.
- Recommendation: Add a blast-radius matrix listing all callers, user-facing impacts, and failure mode per caller; gate rollout by dependency tier (canary subset first, then full).

**[HIGH] Concurrent Cache Refresh and Query Paths Can Race**
- Location: Phase 3; Phase 1 cache management
- Issue: Refresh and search can operate concurrently on the same JSON cache with no locking strategy, risking transient read failures or inconsistent results.
- Evidence: `Background refresh option — ... --refresh-cache` and `save_embedding_cache(cache): Atomic write to cache file`.
- Recommendation: Add file locking around read/write, write-temp+fsync+rename semantics, and retry-on-read-failure; test with simultaneous refresh/search processes.

**[HIGH] Partial Embedding Failures Are Allowed Without Safety Thresholds**
- Location: Error Handling
- Issue: The plan allows silent degradation when embeddings fail, which can return low-quality semantic results without alerting operators.
- Evidence: `Embedding fails | Skip file, continue with others`.
- Recommendation: Define hard thresholds (for example, if failed files >2% then disable semantic results and force keyword fallback), emit structured error metrics, and surface a degraded-mode banner.

**[MEDIUM] Feature Flag Governance Is Incomplete**
- Location: Phase 2 (`--semantic`); Notes
- Issue: The flag exists, but there is no ownership, rollout policy, emergency-disable protocol, or cleanup plan once default behavior changes.
- Evidence: `Add --semantic flag` and `Consider making semantic the default if evaluation proves superiority`.
- Recommendation: Add flag lifecycle policy: owner, default-off/on milestones, kill-switch command, monitoring SLOs, and explicit flag removal date.

### Top 3 Recommendations

1. Add and test a full rollback runbook, including cache downgrade/rebuild paths (Findings: **Rollback Procedure Is Missing**, **Cache Format Has No Versioning**).
2. Make deployment atomic across `cto_context_lib.py` and `cto_context_search.py`, or ship compatibility shims first (Finding: **Deployment Sequence Allows Dependency Mismatch**).
3. Add operational safeguards for runtime safety: lock cache access, enforce failure thresholds, and auto-disable semantic mode when degraded (Findings: **Concurrent Cache Refresh and Query Paths Can Race**, **Partial Embedding Failures Are Allowed**).

### What I Might Be Wrong About

- There may already be an external rollback SOP not included in this document; if so, the rollback findings drop from CRITICAL to documentation-gap severity.
- This script may run only in single-user/local contexts with no concurrent executions; if true, the cache race finding is less severe.
- The implemented code may already include cache version checks and dual-read logic not reflected in the plan text; if true, the schema compatibility finding is overstated.
- The deployment process may always package both files atomically (single commit/single release unit); if true, the dependency-mismatch finding becomes a process documentation issue rather than a release blocker.

## Observability Advocate (codex)

### Findings

**[CRITICAL] Missing Golden Signal Coverage for Semantic Search Path**
- Location: Architecture; Action Items (Phase 1-3); Evaluation Criteria; Success Metrics
- Issue: New production behavior is introduced without explicit latency, traffic, errors, and saturation instrumentation.
- Evidence: "Upgrade `scripts/cto_context_search.py` with optional embedding-based retrieval using local Ollama." and Evaluation Criteria only defines "Token reduction", "Cache efficiency", "Fallback reliability", and "Coverage".
- Recommendation: Add required golden-signal metrics before rollout for each component (`cto_context_search.py`, `cto_context_lib.py`, Ollama API, cache refresh): `request_latency_ms` (p50/p95/p99), `requests_total`, `errors_total` (by error type), and saturation (`queue_depth`, CPU, memory, refresh backlog).

**[CRITICAL] No Alert Thresholds or Paging Policy**
- Location: Evaluation Criteria; Success Metrics
- Issue: The plan defines targets but no warning/page thresholds, so detection is manual and unreliable.
- Evidence: "- [ ] Cache refresh completes in <5 seconds for typical delta" and "- [ ] Zero crashes when Ollama unavailable (graceful degradation)" with no alert definitions.
- Recommendation: Define concrete alert rules with severity tiers and owners, e.g. warn at `semantic_error_rate > 2% for 10m`, page at `>5% for 10m`; warn at `p95_latency > 1.5s`, page at `>2.5s`; page on `embedding_coverage < 99%`.

**[HIGH] Log Strategy Lacks Structured Context and Correlation IDs**
- Location: Error Handling; Verification (Test Results)
- Issue: Current logging approach cannot support fast root-cause analysis during production incidents.
- Evidence: "| Ollama not running | Log warning, fallback to keyword |" and "Warning: Ollama not available. Falling back to keyword search."
- Recommendation: Require structured logs with mandatory fields: `request_id`, `trace_id`, `query_hash`, `mode`, `model`, `cache_hit`, `file_path`, `duration_ms`, `error_code`; reject non-structured warning paths in review.

**[HIGH] Trace Context Breaks at the Ollama Boundary**
- Location: Phase 1 (Embed function); Ollama API
- Issue: Cross-boundary embedding calls have no defined trace propagation, so end-to-end visibility will break.
- Evidence: "`embed_text` — Call Ollama's `/api/embeddings` endpoint" and `curl http://localhost:11434/api/embeddings -d '{ ... }'` (no trace headers defined).
- Recommendation: Propagate `traceparent` and `x-request-id` on every embedding call, log both IDs on caller/callee, and emit span timings for cache lookup, Ollama call, ranking, and fallback decision.

**[HIGH] Partial Failure Mode Is Silent and Undetectable**
- Location: Error Handling
- Issue: Failed embeddings are skipped without a hard health signal, causing silent relevance degradation.
- Evidence: "| Embedding fails | Skip file, continue with others |"
- Recommendation: Track and alert on embedding coverage (`docs_expected`, `docs_embedded`, `skip_rate`), fail refresh when coverage drops below threshold, and expose health status in CLI output and machine-readable metrics.

**[HIGH] Failure Modes Are Listed Without Detection Mechanisms**
- Location: Risks & Mitigations; Notes
- Issue: Risks are documented, but no "how we detect this in production" mechanism is attached.
- Evidence: "| Semantic results worse | Very Low | A/B test before making default |" and "Consider making semantic the default if evaluation proves superiority."
- Recommendation: Add required fields per risk: detection metric, threshold, alert channel, owner, and expected time-to-detect; block default rollout until completed.

**[MEDIUM] Saturation Risk Is Acknowledged but Unmeasured**
- Location: Risks & Mitigations
- Issue: Throughput/latency mitigation is proposed without saturation telemetry, so capacity issues will be found late.
- Evidence: "| Embedding latency too high | Low | Batch files; cache aggressively |"
- Recommendation: Instrument saturation explicitly (`ollama_cpu_pct`, `ollama_mem_mb`, `inflight_embeddings`, `refresh_queue_depth`) and add warn/page thresholds tied to user-visible latency/error impact.

**[MEDIUM] Incident Debug Path Is Incomplete**
- Location: Next Action; Verification; Notes
- Issue: The plan lacks a concrete first-response runbook, so responders will hit "I don’t know what to check next."
- Evidence: "Next Action: `ollama list`" and "Monitor cache size" with no triage sequence for relevance regressions.
- Recommendation: Add an incident runbook for top failure mode (semantic quality drop): first alert, dashboard order, exact commands, expected outputs, decision points, and rollback switch (force keyword-only mode).

### Top 3 Recommendations

1. Add a release gate requiring all four golden signals plus explicit warn/page thresholds and ownership before semantic mode is enabled in production (Findings: 1, 2).
2. Implement end-to-end observability context propagation with structured logs and trace headers across `cto_context_search.py` → Ollama calls (Findings: 3, 4).
3. Convert silent degradation into explicit health signals with embedding coverage metrics, skip-rate alerts, and a runbooked rollback path (Findings: 5, 8).

### What I Might Be Wrong About

- I assumed this document is the full operational spec; if org-level observability standards auto-apply elsewhere, findings 1 and 2 may be overstated.
- I do not have deployment context; if this is strictly local tooling and never production-facing, paging requirements may be too strong.
- I assumed "Log warning" means unstructured logs; if a shared logger injects correlation metadata automatically, finding 3 severity drops.
- Finding 4 may be less severe if Ollama is always local and single-hop and your incident process relies primarily on request-scoped structured logs rather than distributed tracing.

## Coupling Detector (opencode)

### Findings

**[CRITICAL] Global Mutable Cache File Without Versioning**
- Location: `scripts/cto_context_lib.py:461-462`
- Issue: Cache file path is global mutable state with no schema versioning, making format changes dangerous
- Evidence: `CACHE_DIR = ROOT / "logs" / "cache"` and `EMBEDDING_CACHE_FILE = CACHE_DIR / "vault_embeddings.json"` - no version field in cache structure
- Recommendation: Add `"version": "1.0"` field to cache schema; check version on load and rebuild if mismatch

**[CRITICAL] Temporal Coupling in Refresh-Cache Flow**
- Location: `scripts/cto_context_search.py:40-71`
- Issue: `--refresh-cache` uses sys.argv inspection and conditional imports that bypass normal argument parsing, creating fragile initialization order
- Evidence: `if '--refresh-cache' in sys.argv:` and late imports `from cto_context_lib import (...)` inside the branch at lines 44-46
- Recommendation: Move refresh-cache logic to a separate CLI entry point or use subparsers; eliminate sys.argv inspection

**[HIGH] Implicit Cache Structure Contract**
- Location: `scripts/cto_context_lib.py:491-513`
- Issue: Cache structure `{"embedding": [...], "mtime": ..., "title": ...}` is implicit with no validation or documentation
- Evidence: `cache[rel_path] = {"embedding": embedding, "mtime": filepath.stat().st_mtime, "title": ...}` at lines 678-683
- Recommendation: Define a `@dataclass EmbeddingCacheEntry` with validation; serialize with type safety

**[HIGH] Cross-Cutting Ollama Availability Checks**
- Location: `scripts/cto_context_search.py:76-78` and `scripts/cto_context_lib.py:636-643`
- Issue: `ollama_available()` check is duplicated and scattered; if Ollama starts/stops between check and use, the embedding call fails
- Evidence: `if not ollama_available(): print("Warning: Ollama not available...")` then later `embedding = embed_text(text)` without recheck
- Recommendation: Make availability check atomic with the actual embedding call; embed_text should handle unavailability internally

**[MEDIUM] Cache Invalidation on Model Change**
- Location: `scripts/cto_context_lib.py:463`
- Issue: Changing `DEFAULT_EMBED_MODEL` from "nomic-embed-text" invalidates all cached embeddings but cache has no model tracking
- Evidence: `DEFAULT_EMBED_MODEL = "nomic-embed-text"` is constant but not stored in cache; embeddings from different models are incompatible
- Recommendation: Store `"model": "nomic-embed-text"` in each cache entry; `get_stale_files()` should check model mismatch

**[MEDIUM] Fan-In Hotspot on cto_context_lib**
- Location: `grep results showing 10+ scripts importing from cto_context_lib`
- Issue: `cto_context_lib.py` is imported by 10+ scripts; changes to shared utilities risk widespread impact
- Evidence: Imports found in `cto_drift_monitor.py`, `cto_flow.py`, `cto_context_pack.py`, `cto_memory_lookup.py`, `weekly_review.py`, etc.
- Recommendation: Split into focused modules: `search.py`, `cache.py`, `validation.py` to reduce blast radius

**[MEDIUM] Silent Cache Corruption Recovery**
- Location: `scripts/cto_context_lib.py:501-503`
- Issue: Corrupted cache returns empty dict silently, causing full recompute that may surprise users
- Evidence: `except (json.JSONDecodeError, IOError): return {}`
- Recommendation: Log warning on cache corruption with actionable message: "Cache corrupted at {path}, rebuilding..."

**[LOW] Mutable Global Constants**
- Location: `scripts/cto_context_lib.py:17-50`
- Issue: Module-level mutable constants (`SOURCE_WEIGHTS`, `STOPWORDS`) can be modified at runtime by any importer
- Evidence: `SOURCE_WEIGHTS = {...}` and `STOPWORDS = {...}` are plain dict/set at module level
- Recommendation: Use `frozenset()` for STOPWORDS and `MappingProxyType` or tuple pairs for SOURCE_WEIGHTS

### Top 3 Recommendations

1. **Add cache versioning immediately** (addresses CRITICAL finding): Cache format must include version field; on load, validate version and rebuild if mismatch. Prevents silent failures when cache structure changes.

2. **Eliminate sys.argv inspection in refresh-cache** (addresses CRITICAL finding): Use argparse subparsers or separate entry point (`cto_context_refresh.py`). Current approach breaks if flags are reordered or combined.

3. **Make Ollama check atomic with embedding call** (addresses HIGH finding): Merge `ollama_available()` check into `embed_text()`; the function should catch connection errors and return None, eliminating race condition between check and use.

### What I Might Be Wrong About

- **Assumption**: I assumed the cache is accessed by multiple processes simultaneously. If it's single-process only, the atomic write via `.tmp` file is sufficient and my concern about concurrent access is overstated.

- **Lack of context**: I didn't find test files for these modules. If there are comprehensive tests covering cache corruption, model changes, and Ollama unavailability, my severity ratings for those findings might be too high.

- **Finding severity**: The "Fan-In Hotspot" finding might be MEDIUM rather than HIGH if the codebase has strict API stability contracts and integration tests. I rated it HIGH based on import count alone without knowing the team's tolerance for shared library changes.
