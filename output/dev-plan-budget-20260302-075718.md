## Deployment Risk Assessor (deepseek-v3.2)

### Findings

**[CRITICAL] No rollback path for embedding cache corruption**
- Location: Phase 1: Embedding Infrastructure, Phase 3: Incremental Refresh
- Issue: The plan creates a new, critical data file (`vault_embeddings.json`) but provides no mechanism to revert it to a known-good state if the cache becomes corrupted or the embedding generation logic has a bug.
- Evidence: "Cache management — `logs/cache/vault_embeddings.json`" and "Smart caching — Only re-embed files modified since last cache". The `--refresh-cache` command overwrites the entire cache.
- Recommendation: Before any deployment, implement a backup-and-restore mechanism for the cache. The `backup_before_write` function in `cto_context_lib.py` should be called by `save_embedding_cache`. Also, add a `--restore-cache [TIMESTAMP]` CLI command to revert to a previous version.

**[HIGH] Semantic search deployment lacks a feature flag**
- Location: Phase 2: Semantic Search Implementation, Implementation Complete
- Issue: The `--semantic` flag is a user-facing toggle, not a deployment safety control. There is no way to disable the new semantic search functionality at runtime if it causes performance issues or incorrect results, requiring a code rollback.
- Evidence: "Add `--semantic` flag" and "If `--semantic` and Ollama available: ... Else: fall back to keyword search". The feature is live as soon as the code is deployed.
- Recommendation: Introduce a runtime feature flag (e.g., an environment variable `VAULT_SEMANTIC_ENABLED` or a config file setting) that controls whether the `--semantic` flag has any effect. The CLI should respect this flag, defaulting to `False` for the initial deployment. This allows the feature to be turned off without a redeploy.

**[HIGH] Blast radius of failed embedding call is uncontained**
- Location: Phase 2: Semantic Search Implementation, Error Handling table
- Issue: If the embedding of a single file fails during a cache refresh, the plan states "Skip file, continue with others." This leaves the cache in a partially updated, inconsistent state (`mtime` updated but embedding missing or stale), which could cause silent failures in future searches.
- Evidence: "Embedding fails | Skip file, continue with others"
- Recommendation: Implement atomic cache updates. During `--refresh-cache`, build a new, complete cache dictionary in memory. Only call `save_embedding_cache` to overwrite the disk file if the entire process completes successfully. If any file embedding fails, abort the refresh and leave the old cache intact.

**[MEDIUM] No validation for embedding model output format**
- Location: Phase 1: Embedding Infrastructure, `embed_text` function specification
- Issue: The plan assumes the Ollama API will return a 768-dimensional vector, but there is no validation of the response shape or content. A change in the model or API could return malformed data, causing similarity calculations to fail or crash.
- Evidence: "Return 768-dim vector" and the example API response.
- Recommendation: In the `embed_text` function, add validation: check that the response contains a key `"embedding"` that is a list of floats. Log a warning and return `None` if validation fails. The calling code (`search_semantic`, cache refresh) must handle `None` by skipping that item.

**[MEDIUM] Deployment sequence risks service interruption**
- Location: Execution Sequence
- Issue: The deployment plan modifies the shared library (`cto_context_lib.py`) and the CLI script (`cto_context_search.py`) in separate phases but does not specify an order that guarantees zero-downtime for concurrent users or scripts. Deploying the library first could break existing CLI usage if the new functions are missing.
- Evidence: "Day 1: Phase 1 — embedding infrastructure" (library changes), "Day 2-3: Phase 2 — semantic search integration" (CLI changes).
- Recommendation: The deployment must be atomic. Both `cto_context_lib.py` and `cto_context_search.py` should be deployed together in a single operation, as the CLI immediately depends on the new library functions. Use a brief maintenance window or ensure backward compatibility (e.g., new functions are additive, not modifying existing signatures).

**[LOW] Cache TTL configuration is not implemented**
- Location: Phase 3: Incremental Refresh
- Issue: The plan mentions a "Configurable (default: 24 hours)" Cache TTL but the verification shows no implementation of this. Without a TTL, the cache will grow indefinitely as files are deleted or renamed, and stale entries will never be purged.
- Evidence: "Cache TTL — Configurable (default: 24 hours)" in plan, but no reference in Implementation Complete or `cto_context_lib.py` code.
- Recommendation: Implement the TTL. Add a configuration parameter to `get_stale_files` to also return cache entries whose `mtime` is older than a defined TTL (e.g., 24 hours) for re-embedding or deletion. Consider adding a periodic cleanup job.

### Top 3 Recommendations

1. **Implement atomic cache updates and a rollback mechanism** before deploying the cache system. This addresses the CRITICAL finding of no rollback and the HIGH finding of partial failure state. Use the existing `backup_before_write` pattern for the cache file.
2. **Add a runtime feature flag for semantic search** to allow disabling the new functionality without a code rollback, addressing the HIGH risk of a flawed semantic implementation affecting all users.
3. **Deploy library and CLI changes atomically** in a single operation to prevent broken intermediate states, as per the MEDIUM deployment sequence finding.

### What I Might Be Wrong About

- **Assumption about cache criticality**: I'm treating `vault_embeddings.json` as a critical production data store. It might be considered a disposable, rebuildable cache. If it's truly disposable (e.g., can be deleted and fully regenerated with no user impact), the rollback severity is lower.
- **Lack of context on user base**: I assume there are concurrent users or automated processes calling `cto_context_search.py`. If this is only used interactively by a single person, the deployment sequence risk is negligible.
- **Finding on embedding validation might be less severe**: The MEDIUM finding about validating the embedding vector format assumes a crash would be catastrophic. If the calling code (`search_semantic`) robustly handles `None` returns and falls back to keyword search, the impact is just a missing entry, not a service outage.

## Observability Advocate (gemini-3-flash)

### Findings

**[CRITICAL] Missing Saturation Metric for Local Embedding Provider**
- Location: Phase 1: Embedding Infrastructure / Phase 3: Incremental Refresh
- Issue: The plan lacks monitoring for Ollama resource saturation (CPU/VRAM), which can lead to system-wide instability during large cache refreshes.
- Evidence: "Running Ollama locally (nomic-embed-text) means zero API cost... Trade-off: setup friction, model quality variance."
- Recommendation: Implement a concurrency limit and a timeout for the `/api/embeddings` call. Add a check for system load average before initiating a `--refresh-cache` operation to prevent OOM or CPU starvation.

**[HIGH] Broken Trace Correlation in Hybrid Search**
- Location: Phase 2: Semantic Search Implementation / `scripts/cto_context_search.py`
- Issue: There is no shared correlation ID or trace context linking the embedding generation request to the final search result, making it impossible to debug "slow search" incidents in production.
- Evidence: "If --semantic and Ollama available: ... Embed query ... Return top-5".
- Recommendation: Generate a unique `search_id` at the start of `main()`. Pass this ID into `embed_text` and `search_semantic`. Log all timing metrics (embedding latency, similarity calc latency) with this ID.

**[HIGH] Silent Failure Mode in Cache Integrity**
- Location: Phase 1: Embedding Infrastructure / `scripts/cto_context_lib.py`
- Issue: The plan assumes "Atomic write to cache file" is sufficient, but lacks a mechanism to detect or alert on partial/corrupt JSON reads which would break all semantic searches.
- Evidence: "Error Handling: Cache corrupt | Rebuild from scratch".
- Recommendation: Add a checksum (e.g., SHA-256) of the file content to the cache metadata. If the checksum fails on load, trigger a `WARNING` log and automatically initiate a background refresh instead of just failing or silently returning empty results.

**[MEDIUM] Missing Alert Thresholds for Similarity Scores**
- Location: Evaluation Criteria / `scripts/cto_context_search.py`
- Issue: There is no defined "low confidence" threshold for semantic matches, leading to "hallucinated" relevance where the agent might act on low-score noise.
- Evidence: "Return top-5 by cosine similarity + scores".
- Recommendation: Define a `MIN_SIMILARITY_THRESHOLD` (e.g., 0.6). If no results exceed this, the tool should log a `DEBUG` event and explicitly report "No high-confidence semantic matches" to the caller rather than just returning the "least-bad" 5 results.

**[LOW] Log Level Verbosity Risk**
- Location: Phase 3: Incremental Refresh
- Issue: Incremental refresh logs at the same level as search results, potentially cluttering operational logs during frequent file edits.
- Evidence: "print(f" Processed {i}/{len(learning_files)} files...")"
- Recommendation: Move progress indicators to `sys.stderr` or a `DEBUG` log level. Only `INFO` or `ERROR` should go to standard streams to keep the tool's output clean for agent consumption.

### Top 3 Recommendations

1. **Implement Resource Guardrails**: Add a timeout and a "circuit breaker" to the Ollama API client in `cto_context_lib.py`. If the local model takes >2s to respond, the system must automatically trip to keyword-only mode for 5 minutes to prevent cascading latency.
2. **Standardize Structured Logging**: Replace `print` statements in `cto_context_search.py` with structured JSON logging that includes a `correlation_id`, `duration_ms`, and `provider: ollama`. This is essential for identifying if search lag is caused by the model or the similarity math.
3. **Define Confidence Floor**: Set a hard floor for semantic scores (e.g., 0.5). Any result below this should be filtered out before reaching the agent to prevent "semantic drift" where unrelated context is injected into the prompt.

### What I Might Be Wrong About

- **Ollama Performance**: I am assuming Ollama's `/api/embeddings` is the primary bottleneck. If the `numpy` cosine similarity calculation on 100+ vectors is actually slower than the API call, my focus on API timeouts is misplaced.
- **Cache Size**: I rated the cache corruption risk as HIGH. If the vault stays under 500 files, the JSON overhead is negligible and the OS's file write buffers might make "corruption" extremely rare, potentially making this a MEDIUM risk.
- **Agent Behavior**: I assume the agent needs a "No high-confidence matches" signal. If the agent's internal reasoning is robust enough to ignore irrelevant context, then filtering by `MIN_SIMILARITY_THRESHOLD` might actually hide useful "long-tail" connections.

## Coupling Detector (qwen-3.5-27b)

### Findings

**[HIGH] Global ROOT path creates hidden coupling across modules**
- Location: `scripts/cto_context_lib.py:22` and `scripts/cto_context_search.py:44`
- Issue: Both modules depend on `ROOT = Path(__file__).resolve().parents[1]` which assumes a fixed directory structure; moving either file breaks the other
- Evidence: `ROOT = Path(__file__).resolve().parents[1]` in lib; `from pathlib import Path` and `ROOT / "knowledge" / "learnings"` in search script
- Recommendation: Inject ROOT as a parameter to functions that need it, or define it once in a shared config module that both import

**[MEDIUM] Cache file is shared mutable state with no concurrency protection**
- Location: `logs/cache/vault_embeddings.json` referenced in both files
- Issue: Two processes running `--refresh-cache` simultaneously could corrupt the JSON cache file
- Evidence: `save_embedding_cache(cache)` writes atomically but no file locking; `load_embedding_cache()` reads without lock
- Recommendation: Add file locking around cache read/write operations, or use atomic rename pattern (write to temp file, then rename)

**[HIGH] Import statement split creates maintenance coupling**
- Location: `scripts/cto_context_search.py:4-5` and `scripts/cto_context_search.py:44-48`
- Issue: Imports from `cto_context_lib` appear in two separate locations, making it hard to track all dependencies
- Evidence: Top-level imports `search_context, search_semantic, ollama_available`; inside `if '--refresh-cache' in sys.argv:` block imports `load_embedding_cache, save_embedding_cache, _extract_learning_text, embed_text, ROOT`
- Recommendation: Move all imports to top of file; use conditional logic for execution paths instead of conditional imports

**[MEDIUM] Ollama availability check creates temporal coupling on every search**
- Location: `scripts/cto_context_search.py:65-67`
- Issue: `ollama_available()` is called on every semantic search, adding latency even when Ollama is consistently available
- Evidence: `if not ollama_available(): print("Warning: Ollama not available...")` inside the `--semantic` branch
- Recommendation: Cache Ollama availability status with TTL (e.g., 5 minutes) to avoid repeated network checks

**[CRITICAL] `backup_before_write` function exists in lib but is not used anywhere**
- Location: `scripts/cto_context_lib.py:27-41`
- Issue: The plan explicitly calls for shadow restore points (Pattern 3 from Context+ learning), but the function is implemented and never called
- Evidence: `backup_before_write(filepath)` is defined but no call sites exist in `cto_context_search.py` or elsewhere
- Recommendation: Either wire up `backup_before_write()` to be called before cache writes and learning modifications, or remove the dead code

**[LOW] `SOURCE_WEIGHTS` global couples search ranking to hardcoded paths**
- Location: `scripts/cto_context_lib.py:30-37`
- Issue: Adding new directories requires modifying this global dictionary, creating a change amplification point
- Evidence: `SOURCE_WEIGHTS = {"02-validated/patterns/context-packs": 6, "fundamentals": 5, ...}`
- Recommendation: Move to a configuration file or make it loadable from a separate config module

**[MEDIUM] Cache refresh logic assumes sequential file processing without error recovery**
- Location: `scripts/cto_context_search.py:50-63`
- Issue: If embedding fails mid-refresh, the cache is partially updated with no rollback mechanism
- Evidence: Loop processes files one-by-one with `cache[rel_path] = {...}` but no transaction or backup of original cache
- Recommendation: Load original cache, build new cache in memory, then atomically replace; or call `backup_before_write()` on cache file before refresh

**[HIGH] `search_semantic` interface is not versioned and couples callers to implementation**
- Location: `scripts/cto_context_lib.py` (function definition truncated) and `scripts/cto_context_search.py:68-72`
- Issue: Changing `search_semantic` parameters or return type requires updating all callers; no versioning strategy
- Evidence: `search_semantic(args.query, limit=max(1, args.limit), include_transcripts=args.include_transcripts)` — wide parameter list
- Recommendation: Wrap semantic search in a versioned protocol or use a request/response dataclass to encapsulate parameters

### Top 3 Recommendations

1. **Wire up `backup_before_write()` to cache refresh and learning modifications** — This directly addresses the "Shadow Restore Points" pattern from the Context+ learning that motivated this plan. Call it before `save_embedding_cache()` and any vault write operations.

2. **Consolidate all imports to top of `cto_context_search.py`** — Move the conditional imports inside the `--refresh-cache` block to the top of the file. This makes dependencies explicit and prevents future import-related bugs.

3. **Add file locking to cache read/write operations** — Protect `logs/cache/vault_embeddings.json` from concurrent access corruption. Use `fcntl.flock()` on Unix or `msvcrt.locking()` on Windows, or implement atomic rename pattern.

### What I Might Be Wrong About

- **The `backup_before_write` function might be used elsewhere** — I only reviewed the 4 pre-loaded files. If other scripts in the repository call this function, the "dead code" finding is incorrect. However, the plan explicitly lists it as a deliverable for this upgrade, suggesting it should be wired up here.

- **Ollama availability check latency might be negligible** — If `ollama_available()` is a simple TCP connect check that returns in milliseconds, the temporal coupling concern may be overstated. I cannot verify the actual implementation since the function is truncated in the pre-loaded code.

- **The ROOT path coupling might be intentional** — The project may deliberately rely on a fixed directory structure, making the global ROOT acceptable. However, this creates hidden coupling that will surface when someone tries to move or reorganize files, which is a common refactoring scenario.
