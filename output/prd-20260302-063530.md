## Edge Case Hunter (opencode)

I'll analyze the Plan 18 spec using the 5-step security review methodology.
Based on my analysis of the implementation, here is my security review:

---

### Findings

**[CRITICAL] Path Traversal via Cache Keys**
- Location: `save_embedding_cache()`, `load_embedding_cache()`, `search_semantic()`
- Issue: Cache keys are relative paths used directly without validation; malicious cache entries could reference arbitrary files
- Evidence: `rel_path = str(filepath.relative_to(ROOT))` and `cache[rel_path] = {...}` — rel_path stored as-is, later used with `ROOT / rel_path`
- Recommendation: Validate that cached paths resolve within ROOT using `Path.resolve()` and check for `..` components; reject or sanitize paths that escape the vault

**[HIGH] Unbounded Resource Consumption**
- Location: `search_semantic()`, `--refresh-cache` handler
- Issue: No limits on number of files processed or file sizes; can exhaust memory/CPU embedding thousands of large files
- Evidence: `learning_files = [f for f in learnings_path.glob("*.md") ...]` followed by loop embedding all files; `text[:8000]` only truncates post-read
- Recommendation: Add `--max-files` and `--max-file-size` flags with sensible defaults (e.g., 1000 files, 1MB each); skip files exceeding limits with warning

**[HIGH] Race Condition on Cache File**
- Location: `save_embedding_cache()`, concurrent invocations
- Issue: No file locking; simultaneous `--refresh-cache` and `--semantic` search can corrupt cache
- Evidence: `temp_file.replace(EMBEDDING_CACHE_FILE)` is atomic but read/modify/write cycle is not; two processes can interleave
- Recommendation: Use `filelock` or similar for exclusive access during cache operations; or use process ID-based temp files with atomic merge

**[HIGH] TOCTOU on File Modification Check**
- Location: `get_stale_files()`, `search_semantic()`
- Issue: File can change between mtime check and embedding, causing stale data in cache
- Evidence: `actual_mtime = filepath.stat().st_mtime` followed by separate `filepath.read_text()` and `embed_text()` calls
- Recommendation: Open file once, get fd-level mtime with `os.fstat()`, read from same fd; or hash content for change detection

**[MEDIUM] No Embedding Dimension Validation**
- Location: `_cosine_similarity()`, `embed_text()`
- Issue: Assumes 768-dim vectors without validation; mismatched dimensions cause incorrect similarity scores or crashes
- Evidence: `if len(a) != len(b) or len(a) == 0: return 0.0` — only checks length mismatch, not expected dimension
- Recommendation: Validate embedding length equals expected dimension (768 for nomic-embed-text); log error and skip if mismatch

**[MEDIUM] Unicode Injection in Excerpt Display**
- Location: `_extract_learning_text()`, `search_semantic()`
- Issue: No normalization of extracted text; bidirectional characters or homoglyphs could manipulate displayed results
- Evidence: `line.strip()` and direct concatenation with ` | ` used without Unicode normalization or control character filtering
- Recommendation: Apply Unicode normalization (NFC) and strip control characters (except newlines) from extracted text before embedding/display

**[MEDIUM] Unvalidated Ollama URL**
- Location: `embed_text()`, `ollama_available()`
- Issue: Ollama URL can be controlled via parameter; SSRF risk if exposed to external input
- Evidence: `ollama_url: str = DEFAULT_OLLAMA_URL` parameter not validated; could be `http://internal.service:port`
- Recommendation: Validate URL scheme is `http` or `https`, host is `localhost` or `127.0.0.1` unless explicitly allowed

**[MEDIUM] Cache TTL Not Enforced**
- Location: `load_embedding_cache()`, spec section "Phase 3: Incremental Refresh"
- Issue: Spec mentions "Cache TTL — Configurable (default: 24 hours)" but implementation has no TTL enforcement
- Evidence: No timestamp validation in cache entries; embeddings persist indefinitely
- Recommendation: Add `cached_at` timestamp to entries; check age on load; re-embed if older than TTL

**[LOW] Error Information Disclosure**
- Location: `embed_text()`, `ollama_available()`
- Issue: Distinguishable error paths (timeout vs connection refused vs HTTP error) could leak Ollama state
- Evidence: All exceptions return `None` or `False` — actually this is correct behavior, no leakage
- Recommendation: Current behavior is acceptable; no change needed

**[LOW] No Request Rate Limiting**
- Location: `search_semantic()` embedding loop
- Issue: Rapid sequential requests to Ollama API could overwhelm local service
- Evidence: `for filepath in stale_files: ... embedding = embed_text(text, model)` — no delay between requests
- Recommendation: Add small delay (e.g., 0.1s) between embedding requests or batch multiple texts in single request if Ollama supports

---

### Top 3 Recommendations

1. **Implement path validation for all cache operations** — Use `Path.resolve()` and verify paths remain within ROOT before file operations; this prevents the most severe attack vector (arbitrary file access via cache poisoning)

2. **Add file locking and atomic cache updates** — Use `filelock` library to ensure only one process modifies cache at a time; prevents corruption under concurrent access

3. **Enforce resource limits with graceful degradation** — Add `--max-files`, `--max-file-size` limits; skip oversized files with warning rather than attempting to embed them; prevents DoS from unbounded processing

---

### What I Might Be Wrong About

- The cache file permissions might be restricted (e.g., 0600) and the environment single-user, making cache poisoning from external actors unlikely; my race condition findings assume multi-process access that may not occur in practice

- The `text[:8000]` truncation might be intentionally "soft" to handle edge cases gracefully, and the actual Ollama model may have its own limits that prevent abuse; the truncation behavior may be acceptable given Ollama's internal handling

- The TOCTOU vulnerability on file modification might be mitigated by the typical use pattern where files are only modified by the user before running the tool, not during; the race window is small and may not be exploitable in single-user interactive use

- My assessment of Unicode attacks may be overly paranoid for a local CLI tool; the threat model for bidirectional character injection may not apply to this use case where the user controls both input and output display

- The spec mentions "Phase 3: Incremental Refresh" with cache TTL, but the implementation is marked complete; TTL enforcement may be intentionally deferred as noted in "Next Action" items, making this a known limitation rather than an oversight
