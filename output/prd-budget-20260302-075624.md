## Spec Prosecutor (deepseek-v3.2)

### Findings

**[CRITICAL] Undefined "semantic search" behavior when Ollama is partially available**
- Location: Implementation Details / Error Handling
- Issue: The plan specifies fallback to keyword search if Ollama is "not running" or "model not found", but does not define behavior for partial failures (e.g., Ollama responds with HTTP 500, embedding generation times out, or returns malformed vectors).
- Evidence: "Error Handling | Ollama not running | Log warning, fallback to keyword | Model not found | Log warning, fallback to keyword | Embedding fails | Skip file, continue with others"
- Recommendation: Define explicit failure modes and recovery actions: 1) HTTP timeout after N seconds → fallback, 2) Non-200 response → fallback, 3) Malformed JSON or missing 'embedding' key → fallback, 4) Vector dimension mismatch with cache → rebuild cache and retry once.

**[HIGH] Contradiction between "Embedding Target" and actual implementation**
- Location: Implementation Details vs Verification
- Issue: The plan states the embedding target is "Title | Thesis line | Tags", but the verification shows `_extract_learning_text` is used, whose behavior is not defined in the plan. This creates ambiguity about what text is actually embedded.
- Evidence: "Embedding Target: Embed these fields from each learning: - Title (first `# ` heading) - Thesis line (first paragraph after title) - Tags (if present). Concatenate: `title + " | " + thesis + " | " + tags`" vs "Verification: `_extract_learning_text()` is used"
- Recommendation: Either update the plan to specify the exact extraction logic (including handling missing fields) or update the verification to confirm the implemented logic matches the specified concatenation.

**[HIGH] Ambiguous cache staleness definition leads to inconsistent updates**
- Location: Phase 3: Incremental Refresh
- Issue: The plan mentions "Only re-embed files modified since last cache" but does not define "last cache" timestamp. Is it the cache file's mtime, a stored timestamp within the cache, or the latest mtime of any cached entry? This can cause files to be incorrectly skipped or re-embedded.
- Evidence: "Smart caching — Only re-embed files modified since last cache"
- Recommendation: Define the staleness algorithm precisely: "A file is stale if its `os.path.getmtime()` is greater than the `mtime` value stored for its path in the cache dictionary. If the path is not in the cache, it is considered stale."

**[MEDIUM] Boundary condition missing for empty or malformed learning files**
- Location: Phase 1: Embedding Infrastructure
- Issue: The functions `embed_text` and cache management do not specify behavior when a learning file is empty, contains no title, or is unreadable (permission error). The plan says "Embedding fails | Skip file, continue with others" but does not define what constitutes a failure vs. an empty result.
- Evidence: "Embedding fails | Skip file, continue with others"
- Recommendation: Add specification: If `_extract_learning_text` returns an empty string or `None`, the file should be omitted from the cache (or cached with a null embedding and a flag). Permission errors should be logged and the file skipped.

**[MEDIUM] Contradiction in search result ranking and display**
- Location: Phase 2: Semantic Search Implementation vs Evaluation Criteria
- Issue: The plan states semantic search should "Return top-5 by cosine similarity", but the display format shows a markdown table and the evaluation criteria test "Find learnings keyword misses". There is no defined integration policy between semantic and keyword results when both are requested (e.g., via `--semantic` flag).
- Evidence: "Return top-5 by cosine similarity + scores" vs "Display format (add to existing)" showing a combined output, and "Evaluation Criteria | Token reduction | Compare semantic vs keyword for 10 queries"
- Recommendation: Define the combined output specification: When `--semantic` is used, output should clearly separate "Semantic Results" and "Keyword Results" sections (as shown in the verification). Specify that semantic results are ranked by similarity score (0-1), and keyword results retain their original scoring.

**[LOW] Undefined behavior for cache file corruption during read**
- Location: Phase 1: Embedding Infrastructure
- Issue: The plan states "Cache corrupt | Rebuild from scratch" but does not define what constitutes corruption (JSON decode error, missing keys, invalid vector dimensions) or the atomicity of the rebuild operation (e.g., if rebuild fails, is the cache left empty or in previous state?).
- Evidence: "Error Handling | Cache corrupt | Rebuild from scratch"
- Recommendation: Define corruption detection: JSON decode error, missing 'embedding' or 'mtime' keys, or embedding length not equal to 768. On corruption, move the corrupt cache to a backup location, log an error, and rebuild a new cache from scratch. If rebuild fails, the tool should fall back to keyword-only search.

**[LOW] Ambiguous "Coverage" evaluation test**
- Location: Evaluation Criteria
- Issue: The test "Search 'agent loop' | Should find 'adaptive-delegation'" is not a verifiable requirement because it's based on the semantic content of the vault, which may change. This is an example, not a testable contract.
- Evidence: "Coverage | Search 'agent loop' | Should find 'adaptive-delegation'"
- Recommendation: Replace with a testable contract: "For a predefined set of query/concept pairs (e.g., 'agent loop' -> 'adaptive-delegation.md'), semantic search must rank the target learning above a similarity threshold (e.g., >0.6) or within the top N results."

### Top 3 Recommendations

1. **Define partial failure modes for Ollama** (CRITICAL finding). Update the Error Handling table to specify timeout, HTTP error, and malformed response behaviors, ensuring the fallback is robust.
2. **Clarify the embedding text source** (HIGH finding). Align the plan's "Embedding Target" with the implemented `_extract_learning_text` function by documenting the exact concatenation logic in `cto_context_lib.py`.
3. **Specify cache staleness algorithm** (HIGH finding). Replace "modified since last cache" with a precise rule comparing file mtime to a stored mtime within the cache entry.

### What I Might Be Wrong About

- **Assumption about `_extract_learning_text` implementation**: I assumed this function's behavior diverges from the plan's specification. It might actually implement the exact `title + " | " + thesis + " | " + tags` concatenation. If so, the HIGH contradiction finding is less severe.
- **Lack of context on existing error handling**: The `cto_context_lib.py` file was truncated. There might already be robust error handling wrappers for HTTP calls and file I/O that make the CRITICAL finding about partial failures already addressed in the codebase.
- **Severity of the cache corruption finding**: I rated it LOW, but if the cache is large and rebuilding is expensive, a corrupt cache causing a full rebuild could be a performance issue. It might be a MEDIUM risk if the cache takes minutes to rebuild.

## Testability Auditor (gemini-3-flash)

### Findings

**[MEDIUM] Subjective Success Metric for Search Quality**
- Location: Success Metrics
- Issue: The success condition "finds ≥3 relevant learnings" is subjective and cannot be verified by a deterministic test.
- Evidence: "Semantic search finds ≥3 relevant learnings that keyword search misses"
- Recommendation: Define a "Golden Set" of 10 queries and their expected file matches. The test passes if the semantic search recall@5 for these specific queries is ≥ 0.3 higher than keyword search.

**[HIGH] Undefined Cache TTL Logic**
- Location: Phase 3: Incremental Refresh
- Issue: The document specifies a "Cache TTL" of 24 hours but the implementation logic in `cto_context_lib.py` (as described) only uses `mtime` for staleness.
- Evidence: "Cache TTL — Configurable (default: 24 hours)"
- Recommendation: Explicitly state if TTL overrides `mtime`. If a file hasn't changed but the cache is >24h old, does it re-embed? Update `get_stale_files` to include `(current_time - cache_time) > TTL` logic.

**[CRITICAL] Missing Dependency for Cosine Similarity**
- Location: Phase 1: Embedding Infrastructure
- Issue: The plan requires `numpy` for cosine similarity, but the provided `cto_context_lib.py` imports do not include it, and the environment setup is not verified for the runner.
- Evidence: "Cosine similarity — Simple numpy implementation" and "Python dependencies: pip install numpy"
- Recommendation: Add `import numpy as np` to `cto_context_lib.py` and add a verification step in the test suite to check for `numpy` presence before executing semantic tests.

**[LOW] Ambiguous "Thesis Line" Extraction**
- Location: Implementation Details -> Embedding Target
- Issue: The extraction logic for the "thesis line" is heuristic-based and will fail or produce noise on non-standard Markdown files.
- Evidence: "Thesis line (first paragraph after title)"
- Recommendation: Define a fallback behavior. If no paragraph exists after the title, use the first 200 characters of the file or the "Insight" section specifically.

**[MEDIUM] Non-Deterministic "Top 5" Selection on Score Ties**
- Location: Phase 2: Semantic Search Implementation
- Issue: If multiple files return the same cosine similarity (common with short texts or identical tags), the "Top 5" selection is non-deterministic.
- Evidence: "Return top-5 by cosine similarity + scores"
- Recommendation: Specify a secondary sort key for ties, such as alphabetical path name or most recent `mtime`, to ensure test repeatability.

**[HIGH] Integration Gap: Ollama API Timeout/Latency**
- Location: Risks & Mitigations
- Issue: There is no verification strategy for Ollama API latency, which could cause the CLI tool to hang in interactive use.
- Evidence: "Embedding latency too high | Low | Batch files; cache aggressively"
- Recommendation: Add a requirement for a mandatory connection timeout (e.g., 2.0s) on the `urllib` request. A test should verify that if the API hangs, the tool falls back to keyword search within <3 seconds.

### Top 3 Recommendations

1. **Establish a Deterministic Benchmark**: Replace the subjective "relevant learnings" metric with a fixed JSON file of Query-to-Document mappings to allow for automated regression testing of search quality (Relates to Finding 1).
2. **Hard-code API Timeouts**: Implement a strict timeout on the Ollama HTTP call in `cto_context_lib.py`. Without this, the "graceful fallback" is untestable as it depends on the OS-level socket timeout which is often too long (Relates to Finding 6).
3. **Formalize the Staleness Contract**: Clarify the conflict between `mtime` checking and the 24-hour TTL. A deterministic test cannot verify "incremental refresh" if the rules for what constitutes "stale" are contradictory (Relates to Finding 2).

### What I Might Be Wrong About

- **Numpy Necessity**: I assumed `numpy` is strictly required because the plan mentions it. However, cosine similarity for a 768-dim vector can be written in pure Python using `math.sqrt`. If the implementation avoids `numpy`, my "Critical" finding is invalidated.
- **Ollama Availability**: I am assuming Ollama is not pre-installed in the CI environment where these tests would run. If the environment is a local dev machine only, the "Ollama not running" tests are trivial; if it's a CI pipeline, they are complex.
- **Cache Size Impact**: I rated the TTL/Staleness issue as HIGH, but given the small vault size (99 files), a full re-index takes negligible time. The complexity of "incremental" logic might be a premature optimization that doesn't actually impact system reliability.

## Edge Case Hunter (qwen-3.5-27b)

### Findings

**[CRITICAL] Unbounded Query Input Allows Resource Exhaustion**
- Location: Phase 2: Semantic Search Implementation
- Issue: The `--query` parameter has no length limit, allowing attackers to trigger OOM or CPU exhaustion through massive query strings.
- Evidence: `parser.add_argument("--query", required=True, help="Query string.")` with no `type` validation or length constraint
- Recommendation: Add `--query` length limit (e.g., 1000 characters) and validate with `if len(query) > MAX_QUERY_LENGTH: raise ValueError`

**[CRITICAL] Path Traversal in Backup Function**
- Location: `scripts/cto_context_lib.py` - `backup_before_write`
- Issue: The function accepts arbitrary `filepath` without validating it stays within the vault root, enabling backup of sensitive system files.
- Evidence: `def backup_before_write(filepath: str | Path) -> Optional[Path]:` with no path validation against `ROOT`
- Recommendation: Add `src = Path(filepath).resolve(); assert src.is_relative_to(ROOT)` before any file operations

**[HIGH] Cache Corruption Has No Recovery Path**
- Location: Error Handling table
- Issue: Spec states "Cache corrupt → Rebuild from scratch" but provides no mechanism if rebuild fails (e.g., Ollama down, disk full).
- Evidence: `"Cache corrupt | Rebuild from scratch"` in Error Handling table with no fallback
- Recommendation: Implement cache versioning with rollback to previous valid cache, or maintain cache backup before rebuild

**[HIGH] TOCTOU Race Condition in Backup**
- Location: `scripts/cto_context_lib.py` - `backup_before_write`
- Issue: File is read then backed up without locking; concurrent modification between read and copy creates stale backup.
- Evidence: `shutil.copy2(src, backup_path)` without file locking or atomic operations
- Recommendation: Use `fcntl.flock()` on file descriptor or implement atomic copy with temporary file + rename

**[HIGH] No File Locking on Cache Writes**
- Location: Phase 1: Embedding Infrastructure
- Issue: Multiple concurrent searches could corrupt `vault_embeddings.json` through simultaneous writes.
- Evidence: `def save_embedding_cache(cache: Dict): # Atomic write to cache file` - "atomic" claimed but no implementation shown
- Recommendation: Implement file locking with `fcntl.flock()` or use atomic write pattern (write to temp file, then `os.rename()`)

**[MEDIUM] No Timeout on Ollama HTTP Requests**
- Location: Phase 1: Embedding Infrastructure
- Issue: `embed_text()` makes HTTP request to Ollama with no timeout, allowing hung connections to block indefinitely.
- Evidence: `# POST to localhost:11434/api/embeddings` with no timeout parameter specified
- Recommendation: Add `timeout=30` to all `urllib.request.urlopen()` calls with Ollama

**[MEDIUM] No Validation on Embedding Response Format**
- Location: Phase 1: Embedding Infrastructure
- Issue: Code assumes Ollama returns valid 768-dim vector but doesn't validate response structure.
- Evidence: `# Return 768-dim vector` comment with no validation code shown
- Recommendation: Add `assert len(embedding) == 768` and `assert all(isinstance(x, float) for x in embedding)`

**[MEDIUM] No Cache Size Limit**
- Location: Phase 3: Incremental Refresh
- Issue: Cache grows unbounded as files are added; no pruning or size limit specified.
- Evidence: `"Cache grows too large | Low | Compress embeddings; prune old files"` - mitigation is vague
- Recommendation: Add `MAX_CACHE_SIZE = 100MB` and implement LRU eviction or automatic pruning of stale entries

**[MEDIUM] Markdown Encoding Errors Not Handled**
- Location: `scripts/cto_context_lib.py` - `_iter_markdown_files`
- Issue: `path.read_text(encoding="utf-8")` will crash on non-UTF-8 files without error handling.
- Evidence: `lines = path.read_text(encoding="utf-8").splitlines()` with no try/except
- Recommendation: Add `errors="replace"` parameter or wrap in try/except with logging

**[LOW] No Backup Cleanup Policy**
- Location: `scripts/cto_context_lib.py` - `BACKUP_DIR`
- Issue: Backups accumulate indefinitely with no retention policy, risking disk exhaustion.
- Evidence: `BACKUP_DIR = ROOT / "logs" / "backups"` with no cleanup logic
- Recommendation: Add `cleanup_old_backups(max_age_days=30)` function called before each backup

**[LOW] Model Name Not Validated**
- Location: Phase 1: Embedding Infrastructure
- Issue: `model: str = "nomic-embed-text"` accepts arbitrary model names without validation.
- Evidence: `def embed_text(text: str, model: str = "nomic-embed-text")` with no model whitelist
- Recommendation: Add `ALLOWED_MODELS = {"nomic-embed-text"}` and validate `if model not in ALLOWED_MODELS: raise ValueError`

**[LOW] No Rate Limiting on Ollama Calls**
- Location: Phase 2: Semantic Search Implementation
- Issue: No rate limiting on Ollama API calls; concurrent searches could overwhelm local Ollama instance.
- Evidence: No rate limiting mechanism mentioned in spec
- Recommendation: Add `time.sleep(0.1)` between Ollama calls or implement token bucket rate limiter

### Top 3 Recommendations

1. **Implement path validation in `backup_before_write`** (relates to CRITICAL Path Traversal finding) - Add `assert src.is_relative_to(ROOT)` to prevent backup of arbitrary system files
2. **Add file locking to cache operations** (relates to HIGH Cache Corruption finding) - Use `fcntl.flock()` or atomic write pattern to prevent concurrent cache corruption
3. **Add query length limits and timeouts** (relates to CRITICAL Resource Exhaustion finding) - Limit `--query` to 1000 chars and add 30s timeout to all Ollama HTTP requests

### What I Might Be Wrong About

- **Assumption about concurrent usage**: I assumed multiple processes might run searches simultaneously, but if this is a single-user local tool, race conditions may be less critical. If usage is truly single-process, file locking may be unnecessary overhead.

- **Assumption about Ollama reliability**: I rated Ollama failures as HIGH severity, but if the fallback to keyword search is truly robust and Ollama is only used for enhancement (not core functionality), the impact may be lower than I assessed.

- **Assumption about cache corruption likelihood**: I assumed cache corruption is a realistic threat, but if the JSON format is simple and writes are infrequent, corruption may be rare enough that "rebuild from scratch" is acceptable. The severity depends on how often cache rebuilds are needed and how disruptive they are.
